2 Começando

2.1 Insertion sort

Ordenação envolve rearranjar uma sequência de chaves em uma ordem específica. As chaves podem ser qualquer tipo de dado que possa ser comparado para determinar qual é maior ou menor, e geralmente são números. As chaves são frequentemente associadas a outros dados, chamados de dados satélites, que juntos formam um registro (ou objeto).

O insertion sort é um algoritmo eficiente para ordenar um pequeno número de elementos. Ele funciona como a ordenação de um baralho de cartas. Você começa com uma mão esquerda vazia e pega a primeira carta da pilha na mesa. Em seguida, você retira uma carta de cada vez da pilha e a insere na posição correta em sua mão esquerda, comparando-a com cada uma das cartas já presentes na sua mão esquerda. As cartas na sua mão esquerda estão sempre ordenadas.

insertionSort(A, n)
1 for i = 2 to n
2     key = A[i]
3     // Insert A[i] into the sorted subarray A[1 : i - 1].
4     j = i - 1
5     while j > 0 and A[j] > key
6         A[j + 1] = A[j]
7         j = j - 1
8     A[j + 1] = key

Invariante de laço e a corretude do insertion sort

Um invariante de laço é uma propriedade ou condição que é verdadeira antes e depois de cada iteração do laço. Os invariantes de laço nos ajudam a entender por que um algoritmo está correto. Para usar um invariante de laço, você precisa mostrar três coisas:

  • Inicialização: O invariante é verdadeiro antes da primeira iteração do laço.
  • Manutenção: Se for verdadeiro antes de uma iteração, ele permanece verdadeiro antes da próxima iteração.
  • Terminação: O laço termina e o invariante, juntamente com a condição de terminação, nos dá uma propriedade útil que ajuda a mostrar que o algoritmo está correto.

    • Uma prova de invariante de laço é uma forma de indução matemática.

    Vamos aplicar o invariante de laço ao algoritmo insertion sort:

  • Inicialização: A subsequência A[1 : i - 1] está ordenada antes da primeira iteração.
  • Manutenção: O corpo do laço for move os valores para a direita até encontrar a posição correta para A[i], em seguida, insere-o, preservando o invariante de laço.
  • Terminação: O laço termina quando i é igual a n + 1, e a subsequência A[1 : n] consiste nos elementos originalmente em A[1 : n], mas ordenados, o que mostra que o algoritmo está correto.

    2.2 Analisando algoritmos

    Analisar um algoritmo é avaliar a quantidade de recursos que ele utiliza. Esses recursos podem ser o consumo de energia, largura de banda, ou qualquer outra coisa. Mas na maioria das vezes, estamos preocupados em medir o tempo e a memória que um algoritmo consome.

    Usaremos um modelo de computação genérico de random-access machine (RAM). No modelo RAM, as instruções são executadas uma após a outra, sem operações simultâneas.

    Por simplicidade, assumimos que cada instrução leva o mesmo tempo para ser executada - acessar uma variável, modificar uma variável, operações aritméticas, operações lógicas, etc. Isso ocorre porque, no final das contas, para projetar e analisar algoritmos, saber o tempo de cada instrução individual é irrelevante.

    Vários outros modelos computacionais tentam levar em conta os efeitos da hierarquia de memória, que são às vezes significativos em programas reais em máquinas reais. Modelos que incluem a hierarquia de memória são mais complexos que o modelo RAM e podem ser difíceis de trabalhar. No entanto, análises baseadas no modelo RAM geralmente são excelentes preditores de desempenho em máquinas reais.

    Analisar um algoritmo no modelo RAM às vezes pode ser um grande desafio. Você pode precisar usar ferramentas matemáticas, como combinatória, teoria das probabilidades, destreza algébrica e a habilidade de identificar os termos mais significativos em uma fórmula.

    Análise do insertion sort

    Em vez de cronometrar uma execução do insertion sort, podemos determinar quanto tempo ele leva analisando o próprio algoritmo. Vamos examinar quantas vezes cada linha do pseudocódigo é executada.

    O tempo de execução do insertion sort depende da entrada. O fator mais importante é o tamanho da entrada. Dependendo da entrada, o algoritmo pode ter um comportamento de pior caso, de melhor caso, ou de um caso médio.

    A melhor noção para o tamanho da entrada depende do problema em estudo. Para muitos problemas, como ordenação ou transformadas discretas de Fourier, a medida mais natural é o número de itens na entrada. Para muitos outros problemas, como multiplicar dois inteiros, a melhor medida do tamanho da entrada é o número total de bits necessários para representar a entrada em notação binária. Às vezes é mais adequado descrever o tamanho da entrada com mais de um número (grafos).

    O tempo de execução de um algoritmo em uma entrada específica é o número de instruções e acessos a dados executados. Na seguinte análise, cada execução da linha $k$ leva $c_k$ tempo, onde $c_k$ é uma constante. E $t_i$ é o número de vezes que o while da linha 5 é executado para cada $i$.

    insertionSort(A, n)
    		 Custo Quantidade 
    1 for i = 2 to n
    		 $c_1$ $n$ 
    2     key = A[i]
    		 $c_2$ $n-1$ 
    3     // Insert A[i] into the sorted subarray A[1 : i - ].
    		 $0$ $n-1$ 
    4     j = i - 1
    		 $c_4$ $n-1$ 
    5     while j > 0 and A[j] > key
    		 $c_5$ $\sum_{i=2}^nt_i$ 
    6         A[j + 1] = A[j]
    		 $c_6$ $\sum_{i=2}^n(t_i-1)$ 
    7         j = j - 1
    		 $c_7$ $\sum_{i=2}^n(t_i-1)$ 
    8     A[j + 1] = key
    		 $c_8$ $n-1$

    $T(n)=c_1n+c_2(n-1)+c_4(n-1)+c_5\sum_{i=2}^nt_i+c_6\sum_{i=2}^n(t_i-1)+c_7\sum_{i=2}^n(t_i-1)+c_8(n-1)$

    No insertionSort, o melhor caso ocorre quando o array já está ordenado. Portanto, temos $t_i=1$ para cada $i$:

    $\begin{aligned} T(n)&=c_1n+c_2(n-1)+c_4(n-1)+c_5(n-1)+c_8(n-1) \\ &=(c_1+c_2+c_4+c_5+c_8)n-(c_2+c_4+c_5+c_8) \end{aligned}$

    Podemos expressar esse tempo de execução como $an + b$ para constantes $a = c_1 + c_2 + c_4 + c_5 + c_8$ e $b = c_2 + c_4 + c_5 + c_8$ que dependem dos custos das instruções $c_k$. O tempo de execução é, portanto, uma função linear de n.

    O pior caso ocorre quando o array está em ordem reversa. Portanto, $t_i=i$ para cada $i$. Pela soma dos termos de uma PA, temos:

    $\sum_{i=2}^ni=(\sum_{i=1}^ni)-1=\dfrac{n(n+1)}{2}-1$

    e

    $\sum_{i=2}^n(i-1)=\sum_{i=1}^{n-1}i=\dfrac{n(n+1)}{2}$

    O tempo de execução no pior caso é:

    $\begin{aligned} T(n)&=c_1n+c_2(n-1)+c_4(n-1)+c_5(\dfrac{n(n+1)}2-1)+c_6(\dfrac{n(n-1)}2)+c_7(\dfrac{n(n-1)}2)+c_8(n-1) \\ &=(\dfrac {c_5}2 + \dfrac {c_6}2 + \dfrac {c_7}2)n^2+(c_1+c_2+c_4+\dfrac {c_5}2 - \dfrac {c_6}2 - \dfrac {c_7}2 + c_8)n - (c_2+c_4+c_5+c_8) \end{aligned}$

    Podemos expressar esse tempo de execução como $an^2 + bn + c$ para constantes $a = c_5/2 + c_6/2 + c_7/2$, $b = c_1 + c_2 + c_4 + c_5/2 - c_6/2 - c_7/2 + c_8$, e $c = -(c_2 + c_4 + c_5 + c_8)$ que novamente dependem dos custos das instruções $c_k$. O tempo de execução é, portanto, uma função quadrática de n.

    Análise do pior caso e caso médio

    Normalmente nos concentramos em encontrar apenas o tempo de execução no pior caso:

    • O tempo de execução no pior caso de um algoritmo dá um limite superior para o tempo de execução para qualquer entrada. Você tem a garantia de que o algoritmo nunca levará mais tempo.
    • Para alguns algoritmos, o pior caso ocorre com bastante frequência, como encontrar algum dado que não esteja presente no banco de dados.
    • O "caso médio" geralmente é tão ruim quanto o pior caso.

    Em alguns casos particulares, estaremos interessados no tempo de execução de caso médio de um algoritmo. Veremos a técnica da análise probabilística aplicada a vários algoritmos ao longo deste livro. Embora muitas vezes seja difícil definir o que é um "caso médio" para determinados problemas, frequentemente, assumiremos que todas as entradas de um determinado tamanho são igualmente prováveis.

    Ordem de crescimento

    Primeniramente simplificamos a análise ignorando o custo real de cada instrução, usando as constantes $c_k$ para representar esses custos. Também expressamos o tempo de execução de melhor caso como $an + b$ e expressamos o tempo de execução de pior caso como $an^2 + bn + c$.

    É a taxa de crescimento, ou ordem de crescimento, do tempo de execução que realmente nos interessa. Portanto, consideramos apenas o termo principal de uma fórmula, uma vez que os termos de ordem inferior são relativamente insignificantes para valores grandes de $n$. Também ignoramos o coeficiente constante do termo principal. Portanto, para o tempo de execução no pior caso do insertion sort, apenas o fator $n^2$ do termo principal permanece. Por exemplo, suponha que um algoritmo implementado em uma máquina específica leva $n^2/100 + 100n + 17$ microssegundos em uma entrada de tamanho $n$. Embora os coeficientes $1/100$ para o termo $n^2$ e $100$ para o termo $n$ diferem em quatro ordens de magnitude, o termo $n^2/100$ domina o termo $100n$ uma vez que $n$ excede $10.000$. Embora 10.000 possa parecer grande, muitos problemas do mundo real têm tamanhos de entrada muito maiores.

    O insertion sort tem um tempo de execução de pior caso de $\Theta(n^2)$ e um tempo de execução de melhor caso de $\Theta(n)$. Geralmente consideramos um algoritmo mais eficiente que outro se o tempo de execução de pior caso tem uma ordem de crescimento menor: Independentemente dos coeficientes ocultos pela notação $\Theta$, sempre existe algum número tal que um algoritmo de ordem de crescimento menor supera um algoritmo de ordem de crescimento maior; Para valores menores que esse número, geralmente ambos os algoritmos levam tempo razoável.

    2.3 Projetando algoritmos

    O método divide-and-conquer

    O método divide-and-conquer é um algoritmo recursivo que divide um problema em subproblemas menores, resolve-os recursivamente e combina as soluções para criar uma solução para o problema original. O caso base é quando o subproblema é pequeno o suficiente para ser resolvido diretamente, sem recursão. Esse método possui três etapas: dividir, conquistar e combinar.

    O algoritmo merge sort é um exemplo do método divide-and-conquer. Ele ordena um array dividindo-o em dois subarrays, ordenando-os recursivamente e mesclando-os (combinando-os) em um único array ordenado. A operação-chave do algoritmo merge sort é a etapa "combinar", que mescla dois subarrays adjacentes e ordenados.

    O procedimento MERGE funciona escolhendo a menor das duas primeiras cartas de dois montes de cartas ordenados e colocando-a em um monte de saída. Esse processo é repetido até que um dos montes de entrada fique vazio, momento em que o monte de entrada restante é virado sobre o monte de saída. Mesclar demora $\Theta(n)$ tempo.

    MERGE(A, p, q, r)
1 nL = q - p + 1 // length of A[p : q]
2 nR = r - q // length of A[q + 1 : r]
3 let L[0 : nL - 1] and R[0 : nR - 1] be new arrays
4 for i = 0 to nL - 1 // copy A[p : q] into L[0 : nL - 1]
5     L[i] = A[p + i]
6 for j = 0 to nR - 1 // copy A[q + 1 : r] into R[0 : nR - 1]
7     R[j] = A[q + j + 1]
8 i = 0 // i indexes the smallest remaining element in L
9 j = 0 // j indexes the smallest remaining element in R
10 k = p // k indexes the location in A to fill
11 // As long as each of the arrays L and R contains an unmerged element, copy the smallest unmerged element back into A[p : r].
12 while i < nL and j < nR
13     if L[i] ≤ R[j]
14         A[k] = L[i]
15         i = i + 1
16     else A[k] = R[j]
17         j = j + 1
18     k = k + 1
19 // Having gone through one of L and R entirely, copy the remainder of the other to the end of A[p : r].
20 while i < nL
21     A[k] = L[i]
22     i = i + 1
23     k = k + 1
24 while j < nR
25     A[k] = R[j]
26     j = j + 1
27     k = k + 1

    O procedimento MERGE é executado em $\Theta(n)$ tempo, onde $n = r - p + 1$.

    Agora podemos usar o procedimento MERGE como uma subrotina no algoritmo merge sort. $p$ e $r$ são o início e o fim do subarray atual, e $q$ é o ponto médio do subarray. Os subarrays são continuamente divididos até atingirem o tamanho $1$, ou seja, $p=r$.

    mergeSort(A, p, r)
1 if p ≥ r // zero or one element?
2     return
3 q = ⌊(p + r)/2⌋ // midpoint of A[p : r]
4 mergeSort(A, p, q) // recursively sort A[p : q]
5 mergeSort(A, q + 1, r) // recursively sort A[q + 1 : r]
6 // Merge A[p : q] and A[q + 1 : r] into A[p : r].
7 MERGE(A, p, q, r)

    Analisando algoritmos divide-and-conquer

    O tempo de execução de um algoritmo divide-and-conquer pode ser descrito por uma equação de recorrência, que descreve o tempo total de execução em um problema de tamanho $n$ em termos do tempo de execução em entradas menores. Ferramentas matemáticas podem ser usadas para resolver a recorrência e fornecer limites para o desempenho do algoritmo.

    Para tamanhos pequenos de problemas, a solução direta leva tempo constante, representado por $\Theta(1)$. Essa notação ignora o coeficiente do fator 1, assim como $\Theta(n^2)$ ignora o coeficiente do fator $n^2$.

    A divisão de um problema em subproblemas gera $a$ subproblemas, cada um com tamanho $n/b$. O tempo para dividir o problema em subproblemas é $D(n)$ e o tempo para combinar as soluções é $C(n)$, o que resulta na recorrência:

    $\begin{aligned} T(n) &= \begin{cases} \Theta(1) & \text{se } n \lt n_0 \\ D(n) + aT(n/b) + C(n). & \text{caso contrário} \end{cases} \\ \end{aligned}$

    Às vezes, o tamanho da etapa de divisão não é um número inteiro. Por exemplo, o procedimento mergeSort divide um problema de tamanho $n$ em subproblemas de tamanhos $⌈n/2⌉$ e $⌊n/2⌋$. Uma vez que a diferença entre esses tamanhos é pequena, podemos simplificar chamando ambos de tamanho $n/2$. Essa simplificação geralmente não afeta a ordem de crescimento de uma solução para uma recorrência divide-and-conquer.

    Outra convenção é omitir os casos base da recorrência, que geralmente são $T(n) = \Theta(1)$ se $n \lt n_0$ para algum número constante $n_0 \gt 0$. Isso ocorre porque o tempo de execução de um algoritmo em uma entrada de tamanho constante é constante.

    Análise do merge sort

    No mergeSort, a etapa de divisão leva tempo constante, $D(n) = \Theta(1)$. A etapa de conquista resolve recursivamente dois subproblemas, cada um com tamanho $n/2$, contribuindo com $2T(n/2)$ para o tempo de execução. A etapa de combinação leva tempo $\Theta(n)$, $C(n) = \Theta(n)$. O tempo de divisão e combinação do merge sort juntos são $\Theta(n)$. Portanto, o tempo de execução no pior caso do merge sort é:

    $T(n) = 2T(n/2) + cn$.

    Onde a constante $c \gt 0$ representa o tempo necessário para dividir e combinar os subarrays de um array com $n$ itens. Obtemos uma solução exata para a recorrência quando a igualdade é verdadeira e $n$ é uma potência de 2 (dizemos que $n = 2^h$):

    $T(2^h)=2T(2^{h-1})+c2^h$

    Dividindo ambos os lados por $2^h$, temos:

    $T(2^h)/2^h=T(2^{h-1})/2^{h-1}+c$

    Aplicando a mesma equação ao primeiro termo à direita, temos:

    $T(2^h)/2^h=T(2^{h-2})/2^{h-2}+c+c$

    Repetindo o passo anterior $h-2$ vezes, obtemos:

    $T(2^h)/2^h=T(2^0)/2^0+ch$

    Que, após multiplicarmos ambos os lados por $2^h$, nos deixa com a solução:

    $T(n)=T(2^h)=2^h+2^hch=n+cn\lg n=\Theta(n\lg n)$

    O "teorema mestre", uma técnica comum para resolver equações de recorrência que estudaremos em breve, mostra que o tempo de execução no pior caso do merge sort é $\Theta(n \lg n)$ para qualquer $n>0$. Para entradas grandes o suficiente, o merge sort é mais eficiente que o insertion sort no pior caso, cujo tempo de execução é $\Theta(n^2)$.

    1. Usando a Figura 2.4 como modelo, ilustre a operação do merge sort em um array inicialmente contendo a sequência $\langle 3, 41, 52, 26, 38, 57, 9, 49 \rangle$.
    2. O teste na linha 1 do procedimento MERGE-SORT lê "se p ≥ r" em vez de "se p == r". Se o MERGE-SORT for chamado com p > r, o subarray A[p : r] estará vazio. Argumente que, desde que a chamada inicial de MERGE-SORT(A, 1, n) tenha n ≥ 1, o teste "se p == r" é suficiente para garantir que nenhuma chamada recursiva tenha p > r.
    3. Apresente um invariante de laço para o while das linhas 12-18 do procedimento MERGE. Mostre como usá-lo, juntamente com os whiles das linhas 20-23 e 24-27, para provar que o procedimento MERGE está correto.

      4. Um invariante de laço para o while das linhas 12-18 do procedimento MERGE é que, no início de cada iteração do laço, o subarray $A[p : k-1]$ contém ordenados os $k-p$ menores elementos de $L[0 : nL-1]$ e $R[0 : nR-1]$.

      Podemos usar esse invariante de laço, juntamente com os whiles das linhas 20-23 e 24-27, para provar que o procedimento MERGE está correto da seguinte maneira:

      Inicialização: Antes da primeira iteração do laço (quando $k = p$), o subarray $A[p : k-1]$ está vazio, então ele contém ordenados trivialmente os $k-p = 0$ menores elementos de $L[0 : nL-1]$ e $R[0 : nR-1]$.

      Manutenção: Suponha que no início de alguma iteração do laço, $A[p : k-1]$ contém ordenados os $k-p$ menores elementos de $L[0 : nL-1]$ e $R[0 : nR-1]$. Existem dois casos a serem considerados, dependendo se $L[i] \le R[j]$ ou não.

      Se $L[i] \le R[j]$, então $L[i]$ é o menor elemento não incluso (unmerged), então ele é incluído em $A[k]$, e tanto $i$ quanto $k$ são incrementados. Como $L[i]$ é menor ou igual a todos os elementos restantes em $L$ e $R$, segue que após esta iteração, $A[p : k-1]$ contém ordenados os $k-p$ menores elementos de $L[0 : nL-1]$ e $R[0 : nR-1]$.

      Se $L[i] \gt R[j]$, então $R[j]$ é o menor elemento não incluso, então ele é incluído em $A[k]$, e tanto $j$ quanto $k$ são incrementados. Como $R[j]$ é menor ou igual a todos os elementos restantes em $L$ e $R$, segue que após esta iteração, $A[p : k-1]$ contém ordenados os $k-p$ menores elementos de $L[0 : nL-1]$ e $R[0 : nR-1]$.

      Em ambos os casos, o invariante de laço é mantido.

      Terminação: O laço termina quando $i = nL$ ou $j = nR$. Nesse ponto, todos os elementos de um dos arrays $L$ ou $R$ foram incluídos (merged) em $A[p : r]$, então apenas os elementos do outro array permanecem para serem incluídos. Os whiles das linhas 20-23 e 24-27 copiam esses elementos restantes em $A[p : r]$. Como esses elementos restantes são todos maiores ou iguais aos elementos já incluídos em $A[p : r]$, segue-se que no final do procedimento MERGE, $A[p : r]$ contém todos os $nL + nR$ elementos de $L[0 : nL-1]$ e $R[0 : nR-1]$, ordenados.

      Isso completa a prova de que o procedimento MERGE está correto.

      7. recursiveBinarySearch(A, v, low, high)
    if low > high
        return NIL
    mid = floor((low + high) / 2)
    if v == A[mid]
        return mid
    else if v > A[mid]
        return recursiveBinarySearch(A, v, mid + 1, high)
    else return recursiveBinarySearch(A, v, low, mid - 1)

      Cada vez que fazemos a comparação de $v$ com o elemento do meio, o intervalo de busca continua com metade do tamanho. O que nos dá a seguinte recorrência:

      $ T(n) = \begin{cases} \Theta(1) & \text{se $n = 1$,} \\ T(n / 2) + \Theta(1) & \text{se $n > 1$}. \end{cases} $

      A solução da recorrência é $T(n) = T(n/2^1) + \Theta(1) = T(n/2^2) + 2\Theta(1) = T(n/2^3) + 3\Theta(1) = T(n/2^k) + k\Theta(1)$.

      Podemos continuar expandindo a recorrência até alcançarmos o caso base, quando $n/2^k=1$, isto é, $k=\lg n$. Substituindo $k$ na recorrência expandida, temos:

      $T(n)=T(1)+\Theta(1)\lg n$.

      Como $T(1)$ e $\Theta(1)$ são constantes, segue que $T(n)=\Theta(\lg n)$.
