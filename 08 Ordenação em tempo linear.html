8 Ordenao em tempo linear

At o momento, vimos algoritmos que no pior caso custam $O(n\lg n)$. Ser que podemos implementar um algoritmo que no pior caso custe menos que isso?

Os algoritmos de ordenao vistos compartilham uma propriedade interessante: a ordenao feita com base na comparao entre os elementos de entrada. Chamamos tais algoritmos de ordenao por comparao.

8.1 Limites inferiores para ordenao

8.2 Counting sort

Counting sort assume que cada um dos $n$ elementos seja um inteiro entre $0$ e $k$, para algum inteiro $k$. Ele conta a quantidade de elementos menor ou igual a cada elemento, que nos d a posio desse elemento. Quando h elementos repetidos, devemos fazer ajustes para no ca-los todos na mesma posio.

A Figura 8.2 ilustra o counting sort. A descrio do algoritmo :

  1. Inicializa-se um vetor $C[0:k]$ com zero.
  2. Para cada elemento no vetor $A$, seu valor representa o ndice no vetor $C$. Esse valor ento incrementado, resultando em $C$ contendo a contagem de ocorrncias para cada elemento nico em $A$.
  3. Do primeiro ao ltimo elemento de $C$, acumula a contagem de elementos menores que o elemento atual, que so os da sua esquerda.
  4. Do ltimo ao primeiro elemento de $A$, sua posio final ser seu valor correspondente em $C$. Para lidar com duplicatas, o valor em $C$ decrementado aps cada alocao.

O pseudocdigo do algoritmo :

COUNTING-SORT(A, n, k)
 1 let B[1 : n] and C[0 : k] be new arrays
 2 for i = 0 to k
 3     C[i] = 0
 4 for j = 1 to n
 5     C[A[j]] = C[A[j]] + 1
 6 // C[i] now contains the number of elements equal to i.
 7 for i = 1 to k
 8     C[i] = C[i] + C[i - 1]
 9 // C[i] now contains the number of elements less than or equal to i.
10 // Copy A to B, starting from the end of A.
11 for j = n downto 1
12     B[C[A[j]]] = A[j]
13     C[A[j]] = C[A[j]] - 1 // to handle duplicate values
14 return B

Seu tempo de execuo $\Theta(n+k)$. Na prtica, geralmente usamos counting sort quando $k=O(n)$, pois assim seu tempo de execuo $\Theta(n)$.

Counting sort consegue vencer o limite inferior $O(n\lg n)$ de pior caso de ordenao, pois ele no compara os elementos entre si. Na verdade, ele usa os elementos como ndices para outro array.

Uma propriedade importante do counting sort ser estvel: Se antes da ordenao, um elemento $x$ vinha antes de um elemento $y$ com mesma chave, aps a ordenao, $x$ continua antes de $y$.

8.3 Radix sort

Usado antigamente para ordenar cartes de programao com 80 colunas por 12 linhas. Era escolhida uma coluna fixa, e cada carto caia em um de 12 recipientes, conforme seu furo.

Suponha que voc tenha $n$ linhas de nmeros com $d$ colunas de dgitos. Voc pode ordenar uma coluna por vez. Se usar a estratgia da esquerda pra direita (dgitos mais significativos primeiro), ordene recursivamente cada "recipiente".

O radix sort usa a estratgia da direita pra esquerda (dgitos menos significativos primeiro). Ordene a primeira coluna de dgitos, junte tudo, e ordene a segunda coluna, e assim sucessivamente. Para essa estratgia funcionar, a ordenao deve ser estvel. A Figura 8.3 ilustra esse processo com nmeros de 3 dgitos.

Radix sort pode ser usado quando chaves possuem mltiplos campos, como datas compostas pelos campos dia, ms e ano. Intuitivamente, voc poderia seguir a estratgia dos dgitos mais significativos, comear ordenando os anos, e os que empatarem, ordena os meses, e os empates, ordena os dias. Mas se voc seguir a estratgia dos menos significativos, comear ordenando pelos dias, depois pode ordenar tudo novamente pelo ms, e por fim, pelos anos.

O seguinte cdigo do radix sort assume que cada elemento do array possui $d$ dgitos, com o dgito $1$ sendo o menos significativo, e o dgito $d$ sendo o mais significativo.

RADIX-SORT(A,n,d)
1 for i=1 to d
2     use a stable sort to sort array A[1:n] on digit i

Embora no especificado, geralmente usamos o counting sort como algoritmo de ordenao estvel, pois a quantidade $k$ de possveis valores para cada dgito geralmente pequena (para nmeros inteiros, so apenas 10 algarismos). Como o counting sort retorna um novo array, voc pode intercalando o array original e o array cpia a cada iterao, para no ter que ficar repassando da cpia para o original, e ganhar um pouquinho mais de eficincia.

Lema 8.3

Dados $n$ nmeros com $d$ dgitos, cada dgito tendo um de at $k$ possveis valores, radix sort custa $\Theta(d(n+k))$ se o algoritmo de ordenao estvel intermedirio custar $\Theta(n+k)$.

Prova

Cada iterao ordena $n$ nmeros que assumem um valor entre $0$ e $k$, o que custa $\Theta(n+k)$ no counting sort. Como h $d$ iteraes, o custo do radix sort $\Theta(d(n+k))$.

Quando $d$ constante e $k=O(n)$, radix sort executa em tempo linear.

Lema 8.4

Dados $n$ nmeros de $b$ bits e um inteiro positivo $r\le b$, radix sort ordena em tempo $\Theta((b/r)(n+2^r))$ se usar um algoritmo intermedirio de ordenao estvel de custo $\Theta(n+k)$ para elementos no intervalo de $0$ a $k$.

Prova

Para um valor $ r \leq b $, considere cada chave como tendo $ d = \lceil b/r \rceil $ dgitos de $ r $ bits cada. Cada dgito um inteiro no intervalo de 0 a $ 2^r - 1 $, assim, no counting sort, $ k = 2^r - 1 $. (Por exemplo, podemos visualizar uma palavra de 32 bits como tendo quatro dgitos de 8 bits, ento $ b = 32 $, $ r = 8 $, $ k = 2^r - 1 = 255 $ e $ d = b/r = 4 $.) Cada passo do counting sort custa $ \Theta(n + k) = \Theta(n + 2^r) $ e h $ d $ passos, levando a um custo total de $ \Theta(d(n + 2^r)) = \Theta((b/r)(n + 2^r)) $.

Depois o livro entra numa discusso de quais valores ideais para minimizar a expresso, e quando usar radix sort no lugar de um algoritmo como o quicksort. Em suma, quando $b=O(\lg n)$, pode ser interessante usar o radix sort. Mas tambm h outros fatores a se considerar, por exemplo, o quicksort usa o cache do hardware melhor que o radix sort. Alm do mais, o radix sort no ordena in loco, portanto, se uso de memria crtico em sua aplicao, talvez quick sort seja melhor.
