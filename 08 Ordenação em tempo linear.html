8 Ordenação em tempo linear

Até o momento, vimos algoritmos que no pior caso custam $O(n\lg n)$. Será que podemos implementar um algoritmo que no pior caso custe menos que isso?

Os algoritmos de ordenação vistos compartilham uma propriedade interessante: a ordenação é feita com base somente na comparação entre os elementos de entrada. Chamamos tais algoritmos de ordenação por comparação.

8.1 Limites inferiores para ordenação

8.2 Counting sort

Counting sort assume que cada um dos $n$ elementos seja um inteiro entre $0$ e $k$, para algum inteiro $k$. Ele conta a quantidade de elementos menor ou igual a cada elemento, que nos dá a posição desse elemento. Quando há elementos repetidos, devemos fazer ajustes para não caí-los todos na mesma posição.

A Figura 8.2 ilustra o counting sort. A descrição do algoritmo é:

  1. Inicializa-se um vetor $C[0:k]$ com zero.
  2. Para cada elemento no vetor $A$, seu valor representa o índice no vetor $C$. Esse valor é então incrementado, resultando em $C$ contendo a contagem de ocorrências para cada elemento único em $A$.
  3. Do primeiro ao último elemento de $C$, acumula a contagem de elementos menores que o elemento atual, que são os da sua esquerda.
  4. Do último ao primeiro elemento de $A$, sua posição final será seu valor correspondente em $C$. Para lidar com duplicatas, o valor em $C$ é decrementado após cada alocação.

O pseudocódigo do algoritmo é:

COUNTING-SORT(A, n, k)
 1 let B[1 : n] and C [0 : k] be new arrays
 2 for i = 0 to k
 3     C [i] = 0
 4 for j = 1 to n
 5     C [A[j]] = C [A[j]] + 1
 6 // C [i] now contains the number of elements equal to i.
 7 for i = 1 to k
 8     C [i] = C [i] + C [i - 1]
 9 // C [i] now contains the number of elements less than or equal to i.
10 // Copy A to B, starting from the end of A.
11 for j = n downto 1
12     B[C [A[j]]] = A[j]
13     C [A[j]] = C [A[j]] - 1 // to handle duplicate values
14 return B

Seu tempo de execução é $\Theta(n+k)$. Na prática, geralmente usamos counting sort quando $k=O(n)$, pois assim seu tempo de execução é $\Theta(n)$.

Counting sort consegue vencer o limite inferior $O(n\lg n)$ de pior caso de ordenação, pois ele não compara os elementos entre si. Na verdade, ele usa os elementos como índices para outro array.

Uma propriedade importante do counting sort é ser estável: Se antes da ordenação, um elemento $x$ vinha antes de um elemento $y$ com mesma chave, após a ordenação, $x$ continua antes de $y$.

Exercícios

  1. Utilizando a Figura 8.2 como modelo, ilustre a operação do $\text{COUNTING-SORT}$ no array $A = \langle 6, 0, 2, 0, 1, 3, 4, 6, 1, 3, 2 \rangle$.
  2. Prove que $\text{COUNTING-SORT}$ é estável.
  3. Suponha que estivéssemos reescrevendo o cabeçalho do laço for na linha 11 do $\text{COUNTING-SORT}$ como 
    11  para j = 1 até n

    Mostre que o algoritmo ainda funciona corretamente, mas que não é estável. Em seguida, reescreva o pseudocódigo para o counting sort de modo que os elementos com o mesmo valor sejam escritos no array de saída na ordem dos índices crescentes, tornando o algoritmo estável.

    4. O algoritmo é correto, quer $A$ seja processado da frente para trás ou de trás para frente.

    No entanto, o algoritmo modificado não é estável. Como antes, no último laço for, um elemento igual ao retirado de $A$ anteriormente é colocado antes do anterior (ou seja, em uma posição de índice inferior) no array de saída $B$. O algoritmo original era estável porque um elemento retirado de $A$ posteriormente começava com um índice inferior ao de um retirado anteriormente. Mas no algoritmo modificado, um elemento retirado de $A$ posteriormente começa com um índice superior ao de um retirado anteriormente.

    Em particular, o algoritmo ainda coloca os elementos com valor $k$ nas posições $C[k-1] + 1$ até $C[k]$, mas na ordem reversa de sua ocorrência em $A$.

    Poderíamos pensar em estender esse algoritmo colocando os elementos de $A$ em uma coleção de elementos para cada célula no array $C$. Então, se usarmos uma coleção FIFO, a modificação na linha 10 tornará o algoritmo estável; se usarmos uma coleção LILO, ele será antiestável.

  4. Prove a seguinte invariante de laço para o $\text{COUNTING-SORT}$:

    No início de cada iteração do laço for nas linhas 11–13, o último elemento em $A$ com valor $i$ que ainda não foi copiado para $B$ deve estar em $B[C[i]]$.

  5. Suponha que o array sendo ordenado contenha apenas inteiros no intervalo de $0$ a $k$, e que não há dados satélites para serem movidos com essas chaves. Modifique o counting sort para usar apenas os arrays $A$ e $C$, colocando o resultado ordenado de volta no array $A$ em vez de em um novo array $B$.
  6. Descreva um algoritmo que, dado $n$ inteiros no intervalo de $0$ a $k$, realize um pré-processamento em sua entrada e depois responda qualquer consulta sobre quantos dos $n$ inteiros estão dentro de um intervalo $[a..b]$ em tempo $O(1)$. Seu algoritmo deve usar tempo de pré-processamento $\Theta(n + k)$.
  7. Counting sort também pode funcionar eficientemente se os valores de entrada tiverem partes fracionárias, mas o número de dígitos na parte fracionária for pequeno. Suponha que você tenha $n$ números no intervalo de $0$ a $k$, cada um com no máximo $d$ dígitos decimais (base 10) à direita do ponto decimal. Modifique o counting sort para rodar em tempo $\Theta(n + 10^d k)$.
