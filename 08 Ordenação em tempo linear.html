8 Ordenação em tempo linear

Até o momento, vimos algoritmos que no pior caso custam $O(n\lg n)$. Será que podemos implementar um algoritmo que no pior caso custe menos que isso?

Os algoritmos de ordenação vistos compartilham uma propriedade interessante: a ordenação é feita com base na comparação entre os elementos de entrada. Chamamos tais algoritmos de ordenação por comparação.

8.1 Limites inferiores para ordenação

8.2 Counting sort

Counting sort assume que cada um dos $n$ elementos seja um inteiro entre $0$ e $k$, para algum inteiro $k$. Ele conta a quantidade de elementos menor ou igual a cada elemento, que nos dá a posição desse elemento. Quando há elementos repetidos, devemos fazer ajustes para não caí-los todos na mesma posição.

A Figura 8.2 ilustra o counting sort. A descrição do algoritmo é:

  1. Inicializa-se um vetor $C[0:k]$ com zero.
  2. Para cada elemento no vetor $A$, seu valor representa o índice no vetor $C$. Esse valor é então incrementado, resultando em $C$ contendo a contagem de ocorrências para cada elemento único em $A$.
  3. Do primeiro ao último elemento de $C$, acumula a contagem de elementos menores que o elemento atual, que são os da sua esquerda.
  4. Do último ao primeiro elemento de $A$, sua posição final será seu valor correspondente em $C$. Para lidar com duplicatas, o valor em $C$ é decrementado após cada alocação.

O pseudocódigo do algoritmo é:

COUNTING-SORT(A, n, k)
 1 let B[1 : n] and C[0 : k] be new arrays
 2 for i = 0 to k
 3     C[i] = 0
 4 for j = 1 to n
 5     C[A[j]] = C[A[j]] + 1
 6 // C[i] now contains the number of elements equal to i.
 7 for i = 1 to k
 8     C[i] = C[i] + C[i - 1]
 9 // C[i] now contains the number of elements less than or equal to i.
10 // Copy A to B, starting from the end of A.
11 for j = n downto 1
12     B[C[A[j]]] = A[j]
13     C[A[j]] = C[A[j]] - 1 // to handle duplicate values
14 return B

Seu tempo de execução é $\Theta(n+k)$. Na prática, geralmente usamos counting sort quando $k=O(n)$, pois assim seu tempo de execução é $\Theta(n)$.

Counting sort consegue vencer o limite inferior $O(n\lg n)$ de pior caso de ordenação, pois ele não compara os elementos entre si. Na verdade, ele usa os elementos como índices para outro array.

Uma propriedade importante do counting sort é ser estável: Se antes da ordenação, um elemento $x$ vinha antes de um elemento $y$ com mesma chave, após a ordenação, $x$ continua antes de $y$.

8.3 Radix sort

Usado antigamente para ordenar cartões de programação com 80 colunas por 12 linhas. Era escolhida uma coluna fixa, e cada cartão caia em um de 12 recipientes, conforme seu furo.

Suponha que você tenha $n$ linhas de números com $d$ colunas de dígitos. Você pode ordenar uma coluna por vez. Se usar a estratégia da esquerda pra direita (dígitos mais significativos primeiro), ordene recursivamente cada "recipiente".

O radix sort usa a estratégia da direita pra esquerda (dígitos menos significativos primeiro). Ordene a primeira coluna de dígitos, junte tudo, e ordene a segunda coluna, e assim sucessivamente. Para essa estratégia funcionar, a ordenação deve ser estável. A Figura 8.3 ilustra esse processo com números de 3 dígitos.

Radix sort pode ser usado quando chaves possuem múltiplos campos, como datas compostas pelos campos dia, mês e ano. Intuitivamente, você poderia seguir a estratégia dos dígitos mais significativos, começar ordenando os anos, e os que empatarem, ordena os meses, e os empates, ordena os dias. Mas se você seguir a estratégia dos menos significativos, começar ordenando pelos dias, depois pode ordenar tudo novamente pelo mês, e por fim, pelos anos.

O seguinte código do radix sort assume que cada elemento do array possui $d$ dígitos, com o dígito $1$ sendo o menos significativo, e o dígito $d$ sendo o mais significativo.

RADIX-SORT(A,n,d)
1 for i=1 to d
2     use a stable sort to sort array A[1:n] on digit i

Embora não especificado, geralmente usamos o counting sort como algoritmo de ordenação estável, pois a quantidade $k$ de possíveis valores para cada dígito geralmente é pequena (para números inteiros, são apenas 10 algarismos). Como o counting sort retorna um novo array, você pode intercalando o array original e o array cópia a cada iteração, para não ter que ficar repassando da cópia para o original, e ganhar um pouquinho mais de eficiência.

Lema 8.3

Dados $n$ números com $d$ dígitos, cada dígito tendo um de até $k$ possíveis valores, radix sort custa $\Theta(d(n+k))$ se o algoritmo de ordenação estável intermediário custar $\Theta(n+k)$.

Prova

Cada iteração ordena $n$ números que assumem um valor entre $0$ e $k$, o que custa $\Theta(n+k)$ no counting sort. Como há $d$ iterações, o custo do radix sort é $\Theta(d(n+k))$.

Quando $d$ é constante e $k=O(n)$, radix sort executa em tempo linear.

Lema 8.4

Dados $n$ números de $b$ bits e um inteiro positivo $r\le b$, radix sort ordena em tempo $\Theta((b/r)(n+2^r))$ se usar um algoritmo intermediário de ordenação estável de custo $\Theta(n+k)$ para elementos no intervalo de $0$ a $k$.

Prova

Para um valor $ r \leq b $, considere cada chave como tendo $ d = \lceil b/r \rceil $ dígitos de $ r $ bits cada. Cada dígito é um inteiro no intervalo de 0 a $ 2^r - 1 $, assim, no counting sort, $ k = 2^r - 1 $. (Por exemplo, podemos visualizar uma palavra de 32 bits como tendo quatro dígitos de 8 bits, então $ b = 32 $, $ r = 8 $, $ k = 2^r - 1 = 255 $ e $ d = b/r = 4 $.) Cada passo do counting sort custa $ \Theta(n + k) = \Theta(n + 2^r) $ e há $ d $ passos, levando a um custo total de $ \Theta(d(n + 2^r)) = \Theta((b/r)(n + 2^r)) $.

Depois o livro entra numa discussão de quais valores ideais para minimizar a expressão, e quando usar radix sort no lugar de um algoritmo como o quicksort. Em suma, quando $b=O(\lg n)$, pode ser interessante usar o radix sort. Mas também há outros fatores a se considerar, por exemplo, o quicksort usa o cache do hardware melhor que o radix sort. Além do mais, o radix sort não ordena in loco, portanto, se uso de memória é crítico em sua aplicação, talvez quick sort seja melhor.
