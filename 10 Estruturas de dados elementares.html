Estruturas de Dados

Introdução

Enquanto os conjuntos matemáticos são invariáveis, os conjuntos manipulados por algoritmos podem crescer, encolher ou sofrer outras mudanças ao longo do tempo. Chamamos tais conjuntos de conjuntos dinâmicos.

Damos o nome de dicionário ao conjunto dinâmico em que cada elemento, chamado de chave, possui um valor associado. Esse nome vem do termo dicionário, em que procuramos pelas definições (valor) das palavras (chave). Um outro exemplo de uma estrutura do tipo “dicionário” seria o índice de um livro, que lista as palavras-chaves (chave), com as páginas em que ocorrem (valor).

O elemento pode conter dados satélites, que são outros atributos além da chave que não são utilizados pela implementação do conjunto.

Apresentamos a seguir, uma lista de operações típicas em conjuntos dinâmicos:

  • search(S, k) - Uma consulta que, dado um conjunto S e uma chave k, retorna um ponteiro x para um elemento em S tal que x.key = k ou NIL se nenhum elemento desse tipo pertencer a S.
  • insert(S, x) - Uma operação modificadora que aumenta o conjunto S com o elemento apontado por x.
  • delete(S, x) - Uma operação modificadora que, dado um ponteiro x para um elemento no conjunto S, exclui x de S. (Observe que essa operação utiliza um ponteiro para um elemento x, não um valor de chave.)
  • mininum(S) - Uma consulta em um conjunto ordenado S que retorna um ponteiro para o elemento de S que tenha a menor chave.
  • maximum(S) - Uma consulta em um conjunto ordenado S que retorna um ponteiro para o elemento de S que tenha a maior chave.
  • successor(S, x) - Uma consulta que, dado um elemento x cuja chave é de um conjunto ordenado S, retorna um ponteiro para o menor elemento após x em S ou NIL se x é o elemento máximo.
  • predecessor(S, x) - Uma consulta que, dado um elemento x, cuja chave é de um conjunto ordenado S, retorna um ponteiro para o maior elemento antes de x em S ou NIL se x é o elemento mínimo.

Estruturas de dados elementares

10.1 Estruturas de dados baseadas em array

10.1.1 Arrays

Um array é armazenado como uma sequência contígua de bytes na memória. Devido a esse fato, é possível calcular em tempo constante a localização de um elemento na memória sabendo o seu índice. Isto é, leva tempo constante acessar qualquer elemento pelo seu índice.

Isso implica que os elementos do array devem ocupar o mesmo espaço na memória. Quando não for o caso, o array armazena endereços de memória que apontam para objetos, que já não ficam armazenados diretamente no array. Ainda assim, o acesso a um objeto, sabendo seu índice no array, continua constante.

10.1.2 Matrizes

Tipicamente representamos matrizes como arrays bidimensionais. Mas também podemos ter matrizes multidimensionais. Como a memória é basicamente uma sequência contígua de bytes, matrizes de fato são armazenadas como arrays, e ao acessar um elemento através dos índices de linha e coluna, o computador faz uma aritmética para saber o índice na memória.

$$ \begin{pmatrix} \begin{array}{c|c} \begin{matrix} 1 & 2 \\ 5 & 6 \end{matrix} & \begin{matrix} 3 & 4 \\ 7 & 8 \end{matrix} \\ \hline \begin{matrix} 9 & 12 \\ 13 & 14 \end{matrix} & \begin{matrix} 11 & 12 \\ 15 & 16 \end{matrix} \end{array} \end{pmatrix} $$

pode ser armazenada em um único array na ordem 〈1, 2, 5, 6, 3, 4, 7, 8, 9, 10, 13, 14, 11, 12, 15, 16〉.

Pilhas e filas

Numa pilha, o elemento inserido vai para o topo da pilha, e só pode ser excluído o elemento que está no topo. Essa política de exclusão é chamada de LIFO, ou last in, first out. Numa fila, o elemento inserido vai para o fim da fila, e só pode ser excluído o elemento que está no início da fila. Já essa política é FIFO, ou first in, first out.

Pilhas

A Figura 10.2 ilustra a implementação de uma pilha com no máximo n elementos em um array S[1:n]:

Quando S.top=0, a pilha está vazia. Não precisamos zerar as posições do array, consideramos o que está lá como “lixo”. Tentar inserir um elemento numa pilha cheia ocorre stack overflow (estouro de pilha, ou transbordamento de pilha), e tentar excluir um elemento de uma pilha vazia ocorre stack underflow (escassez de pilha, ou esgotamento de pilha).

STACK-EMPTY(S)
1 if S.top == 0
2      return TRUE
3 else return FALSE

PUSH(S, x)
1 if S.top == S.size
2      error “overflow”
3 else S.top = S.top + 1
4      S[S.top] = x

POP(S)
1 if STACK-EMPTY(S)
2      error “underflow”
3 else S.top = S.top - 1
4      return S[S.top + 1]

Todas as operações têm custo constante.

Filas

A Figura 10.3 ilustra a implementação de uma fila com no máximo n-1 elementos em um array Q[1:n]:

A fila segue uma ordem circular. Quando Q.head==Q.tail, a fila está vazia. Quando Q.head==Q.tail+1 ou Q.head==1 e Q.tail=Q.size, a fila está cheia. Na fila também pode ocorrer underflow ou overflow. O seguinte código não evita underflow e overflow, que são deixados como exercícios.

ENQUEUE(Q, x)
1 Q[Q.tail] = x
2 if Q.tail == Q.size
3      Q.tail = 1
4 else Q.tail = Q.tail + 1

DEQUEUE(Q)
1 x = Q[Q.head]
2 if Q.head == Q.size
3      Q.head = 1
4 else Q.head = Q.head + 1
5 return x

Todas as operações têm custo constante.

10.2 Listas encadeadas

Em listas encadeadas, ao invés dos elementos ocuparem posições contíguas na memória como em arrays, os elementos são apontados uns pelos outros consecutivamente. Também são chamadas de listas ligadas.

A Figura 10.4 ilustra uma lista duplamente encadeada, em que cada nó aponta tanto para o próximo nó quanto para o nó anterior.

Elementos são objetos que contém o campo key e os demais dados satélites. A forma de implementação da lista pode considerar o nó sendo o elemento, ou, considerar o nó contendo o elemento. Por simplificação, no livro, o nó será o elemento.

Na lista encadeada simples, cada nó aponta apenas para o próximo nó, sem apontar para o anterior. Na lista encadeada circular o nó anterior a head é o último nó da lista. A lista pode ser ordenada ou não, o que afeta a posição em que os nós podem ser inseridos.

Buscando em uma lista encadeada

Algoritmo:

LIST-SEARCH(L, k)
1 x = L.head
2 while x ≠ NIL and x.key ≠ k
3   x = x.next
4 return x

O custo é $\Theta(n)$ no pior caso, quando o elemento desejado está no final da lista.

Inserindo em uma lista encadeada

Algoritmo de inserção no início:

LIST-PREPEND(L, x)
1 x.next = L.head
2 x.prev = NIL
3 if L.head ≠ NIL
4   L.head.prev = x
5 L.head = x

O custo é $\Theta(1)$.

Algoritmo de inserção de um novo nó x na frente de um nó y:

LIST-INSERT(x, y)
1 x.next = y.next
2 x.prev = y
3 if y.next ≠ NIL
4   y.next.prev = x
5 y.next = x

Como já temos o nó y, o custo de inserção é $\Theta(1)$. Mas para buscarmos y pela chave, o custo de pior caso é $\Theta(n)$.

Exclusão em uma lista encadeada

Algoritmo de exclusão de um nó x:

LIST-DELETE(L, x)
1 if x.prev ≠ NIL
2      x.prev.next = x.next
3 else L.head = x.next
4 if x.next ≠ NIL
5      x.next.prev = x.prev

Como já temos o nó x, o custo de exclusão é $\Theta(1)$. Mas para buscarmos x pela chave, o custo de pior caso é $\Theta(n)$.

Sentinela

Uma sentinela é um nó que representa NIL. O primeiro e o último nó da lista, ao invés de apontar para NIL, aponta para a sentinela. Nessa configuração, não precisamos mais de uma referência a uma lista com seu campo head, apenas à sentinela. E a lista se torna duplamente encadeada circular. A Figura 10.5 ilustra a lista com sentinela:

Na Figura 10.5, a sentinela é representada pelo nó azul, e fica entre o head e o tail.

Algoritmo de exclusão adaptado:

LIST-DELETE′(x)
1 x.prev.next = x.next
2 x.next.prev = x.prev

Agora podemos ignorar os limites de head e tail da lista. Só temos que cuidar para não excluir a sentinela, pois assim perderíamos a referência para a lista.

Algoritmo de inserção adaptado:

LIST-INSERT′(x, y)
1 x.next = y.next
2 x.prev = y
3 y.next.prev = x
4 y.next = x

Agora para inserir no início (head), y é a sentinela, e para inserir no fim (tail), y é o nó anterior à sentinela.

Nessas adaptações economizamos as intruções condicionais mas os algoritmos continuam com o mesmo custo assintótico $O(1)$.

Algoritmo de busca adaptado:

LIST-SEARCH′(x, k)
1 x = x.next // start at the head of the list
2 while x.key ≠ NIL and x.key ≠ k
3      x = x.next
4 if x.key == NIL // did not find k
5      return NIL
6 else return x // found k in element x

Para diferenciar a sentinela dos demais nós, consideramos que sua chave é NIL, isto é, nenhum outro nó pode ter essa mesma chave. Além do mais, esse algoritmo considera que x inicialmente seja a sentinela, para garantir que todos os elementos sejam averiguados.

Nessa adaptação, acrescentamos algumas instruções mas o algoritmo continua com o mesmo custo assintótico $\Theta(n)$.
