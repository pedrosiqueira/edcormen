11 Tabelas hash

Muitas aplicaes exigem um conjunto dinmico que suporte somente as operaes de dicionrio INSERT, SEARCH e DELETE. Por exemplo, um compilador que traduz uma linguagem de programao mantm uma tabela de smbolos. 11.1 Tabelas de endereamento direto

O endereamento direto uma tcnica simples que funciona bem quando o universo $U$ de chaves razoavelmente pequeno. Suponha que uma aplicao necessite de um conjunto dinmico no qual cada elemento tem uma chave extrada do universo $U = \{0, 1, ..., m - 1\}$, onde $m$ no muito grande. Consideramos que no h dois elementos com a mesma chave.

Para representar o conjunto dinmico, usamos um array, ou uma tabela de endereamento direto, denotada por $T[0 .. m - 1]$, na qual cada posio (ou lacuna, slot) corresponde a uma chave no universo $U$. A Figura 11.1 ilustra a abordagem; a posio $k$ aponta para um elemento no conjunto com chave $k$. Se o conjunto no contm nenhum elemento com chave $k$, ento $T[k] = NIL$.

A implementao das operaes de dicionrio trivial:

directAddressSearch(T,k)
    return T[k]

directAddressInsert(T,x)
    T[x.key] = x

directAddressDelete(T,x)
    T[x.key] = NIL

Cada uma dessas operaes custa somente $O(1)$.

Em vez de armazenar o objeto fora da tabela de endereamento direto, podemos armazenar o objeto na prpria tabela. Essa alternativa economizaria espao? Usaramos uma chave especial dentro do objeto para indicar uma posio vazia. Alm disso, muitas vezes, desnecessrio armazenar a chave do objeto, j que, se temos o ndice de um objeto na tabela, temos sua chave.

11.2 Tabelas hash

Se o universo $U$ grande, armazenar uma tabela $T$ de tamanho $|U|$ pode ser impraticvel ou mesmo impossvel. Alm disso, o conjunto $K$ de chaves realmente armazenadas pode ser to pequeno em relao a $U$ que grande parte do espao alocado para $T$ seria desperdiada.

Com tabela de endereamento direto, um elemento com a chave $k$ armazenado na posio $k$. Com tabela hash (tabela de espalhamento, ou tabela de disperso), esse elemento armazenado na posio $h(k)$; isto , usamos uma funo hash $h$ para calcular a posio da chave $k$.

Hashing uniform independente

Para cada elemento $k$ do domnio $U$, uma funo hash $h$ ideal geraria o mesmo valor $h(k)$ independentemente e uniformemente aleatrio do intervalo $\{0, 1, ..., m - 1\}$. Chamamos tal funo de funo hash uniforme independente. Na prtica, temos aproximaes desse ideal.

Resoluo de colises por encadeamento

No encadeamento, todos os elementos resultantes do hash vo para a mesma posio em uma lista encadeada, como mostra a Figura 11.3.

Se a lista for duplamente encadeada, a operao de excluso pode ser feita em tempo $O(1)$, j que o argumento da funo no a chave do elemento a ser excludo, e sim o ponteiro para o elemento. A insero $O(1)$.

Anlise do hashing com encadeamento

Dada uma tabela hash $T$ com $m$ posies que armazena $n$ elementos, definimos o fator de carga $\alpha$ de $T$ como $n/m$, isto , o nmero mdio de elementos armazenados em uma lista encadeada.

O pior caso do hashing com encadeamento terrvel: aps o hashing, todas as $n$ chaves vo para a mesma posio, criando uma lista de comprimento $n$.

O desempenho do hashing para o caso mdio depende de como a funo hash $h$ distribui as chaves entre as $m$ posies, em mdia. Por enquanto, vamos considerar hashing uniform independente, isto , qualquer elemento tem igual probabilidade de cair em qualquer das $m$ posies, independente dos elementos j na tabela.

Para $j = 0, 1, ..., m - 1$, vamos denotar o comprimento da lista $T[j]$ por $n_j$, de modo que

$$\begin{equation} n=n_0+n_1+...+n_{m-1} \end{equation}$$

e o valor esperado de $n_j$ $E[n_j] = \alpha = n/m$.

Assumimos que o custo de calcular o valor hash $h(k)$ $O(1)$. Assim, o tempo para procurar por um elemento com chave $k$ depende do comprimento $n_{h(k)}$ da lista $T[h(k)]$.

Teorema 11.1

Em uma tabela hash com colises resolvidas com encadeamento, uma busca malsucedida demora tempo mdio $\Theta(1+\alpha)$ com hashing uniforme independente.

Prova

Qualquer chave $k$ ainda no armazenada na tabela tem igual probabilidade de ocupar qualquer das $m$ posies aps o hash. O tempo esperado para procurar sem sucesso uma chave $k$ o tempo esperado para buscar at o fim da lista $T[h(k)]$, que tem o comprimento esperado $E[n_{h(k)}] = \alpha$. Assim, o nmero esperado de elementos examinados em uma busca malsucedida $\alpha$, e o tempo total exigido (incluindo o tempo para se calcular $h(k)$) $\Theta(1 + \alpha)$.

Teorema 11.2

Em uma tabela hash com colises resolvidas com encadeamento, uma busca bem sucedida demora tempo mdio $\Theta(1+\alpha)$ com hashing uniforme independente.

Prova

A situao para uma busca bem-sucedida ligeiramente diferente, pois o elemento desejado pode ser qualquer um dos elementos presentes na tabela, de modo que quanto maior uma lista, maior a probabilidade da busca cair nela. Todavia, o tempo de busca esperado ainda $\Theta(1 + \alpha)$. Mais detalhes sobre a prova encontra-se no livro.

O que estas anlises significam? Se o nmero de posies da tabela hash no mnimo proporcional ao nmero de elementos na tabela, temos $n = O(m)$ e, consequentemente, $\alpha = n/m = O(m)/m = O(1)$. Assim, a busca demora tempo constante na mdia.

11.3 Funes hash

Uma boa funo hash satisfaz (aproximadamente) a premissa do hashing uniforme independente: cada chave tem igual probabilidade de cair em qualquer das $m$ posies, independentemente da posio que qualquer outra chave ocupa. Infelizmente, normalmente no temos nenhum meio de verificar essa condio, j que raramente conhecemos a distribuio de probabilidade da qual as chaves so extradas. Alm disso, as chaves poderiam no ser extradas independentemente. Ocasionalmente conhecemos a distribuio. Por exemplo, se soubermos que as chaves so nmeros reais aleatrios $k$, independente e uniformemente distribudos na faixa $0 \leq k \lt 1$, ento a funo hash $h(k)=\lfloor km \rfloor$ satisfaz a condio de hashing uniforme independente.

A maior parte das funes hash considera como universo de chaves o conjunto $N = \{0, 1, 2, ...\}$ de nmeros naturais. Assim, se as chaves no forem nmeros naturais, temos de encontrar um modo de interpret-las como nmeros naturais. Por exemplo, o conjunto de caracteres ASCII possui 128 possveis caracteres. Para converter uma string ASCII (dgitos na base 128) em um nmero natural (dgitos na base 10), usamos a frmula de converso de radix. Assim, a string pt (caracteres 112 e 116) na notao decimal seria 112*128+126=14452. Como seria teu nome na notao decimal? Portanto, consideramos a partir daqui que as chaves so nmeros naturais.

11.3.1 Hashing esttico

O mtodo da diviso

Funo hash: $h(k) = k \mod m$.

Por exemplo, para tabela hash com $m = 12$ e chave $k = 100$, ento $h(k) = 4$.

Vantagem: rapidez, pois o clculo envolve apenas uma operao de diviso.

Alguns valores para $m$ so ruins, como uma potncia de $2$, j que, se $m = 2^p$, ento $h(k)$ usar apenas os $p$ bits menos significativos de $k$ para o clculo. melhor garantir que a funo hash dependa de todos os bits da chave.

Um primo no muito prximo de uma potncia de $2$ uma boa escolha para $m$. Por exemplo, considere uma tabela hash com colises resolvidas por encadeamento que armazena $n=2000$ strings. Um bom tamanho para essa tabela $701$, pois um primo no prximo de uma potncia de $2$ e permite uma mdia de trs elementos colididos ($2000/3 701$). A funo hash seria $h(k)=k \mod 701$.

O mtodo da multiplicao

Funo hash:

$$h(k) = \lfloor m (k A \mod 1)\rfloor$$

que pode ser lida da seguinte forma:

  1. Escolha uma constante $A$ no intervalo $0 \lt A \lt 1$
  2. Multiplique a chave $k$ pela constante $A$
  3. Extraia a parte fracionria de $k A$, isto , $(k A \mod 1)$ o mesmo que $(k A - k A)$
  4. Multiplique a parte fracionria por $m$
  5. Pegue o piso do resultado

Vantagem: o valor de $m$ no crtico. Entretanto, este mtodo no to rpido quanto o da diviso. Em geral, $m = 2^p$, para algum inteiro $p$, j que podemos implementar facilmente a funo na maioria dos computadores (mais informaes no livro).

Alguns valores para $A$ funcionam melhor que outros. A escolha tima depende das caractersticas dos dados aos quais o hash est sendo aplicado. Knuth sugere que $A \approx ( \sqrt{5} - 1 )/2=0.6180339887...$. funciona bem.

Como exemplo, suponha $k=123456$, $p=14$, $m=2^p=16384$. Assim, $kA = 76300.004115107$. Ento, $h(k) = m * 0.004115107 = 67$.

O mtodo da multiplicao com deslocamento

Se as chaves couberem em uma palavra de mquina de $w$ bits, e $m=2^l\le w$, podemos usar o mtodo da multiplicao com deslocamento (multiply-shift).

Seja a constante $a=A2^w$, onde $0\le A \le 1$, de modo que $0\le a \le 2^w$, a funo hash :

$$\begin{equation}h_a(k) = (ka \mod 2^w) \gg (w-l)\end{equation}$$

que pode ser lida assim:

  1. Multiplique $k$ pelo inteiro $a$ de $w$ bits. O resultado um valor com $2w$ bits.
  2. Pegue o resto da diviso por $2^w$, eliminando os $w$ bits mais significativos ($r_1$), deixando apenas o $w$ bits menos significativos ($r_0$).
  3. Desloque (shift) direita $w-l$ bits, deixando apenas os $l$ bits mais significativos de $r_0$, que seu valor hash.

11.3.2 Hashing aleatrio

Suponha que um cracker escolha chaves sabendo a funo hash esttica implementada. Ele pode escolher $n$ chaves que caiam no mesmo slot, resultando em um tempo mdio de busca de $\Theta(n)$. Qualquer funo hash esttica vulnervel a esse ataque. A nica maneira eficaz de evitar essa situao escolher a funo hash aleatoriamente e independentemente das chaves. Essa abordagem chamada de hashing aleatrio. Um caso especial dessa abordagem, chamado de hashing universal, pode proporcionar um desempenho mdio comprovadamente bom, independentemente das chaves escolhidas pelo cracker.

Para usar hashing aleatrio, no incio da execuo do programa, escolha aleatoriamente uma funo hash de uma famlia adequada de funes. Assim como no quicksort, aleatoriedade garante que uma sequncia de entrada no invoque apenas o pior caso a cada execuo.

Por exemplo, uma variante aleatria simples do mtodo de multiplicao com deslocamento que funciona bem na mdia , no incio do programa, escolher aleatoriamente um nmero mpar para o valor de $a$.

Uma famlia $H$ de funes hash universal se, ao escolher aleatoriamente uma funo hash $h$, a chance de coliso entre duas chaves arbitrrias distintas $k_1,k_2$ for no mximo a chance $1/m$ da coliso se $h(k_1)$ e $h(k_2)$ fossem valores aleatoriamente e independentemente escolhidos entre $\{0,1,..., m-1\}$.

Hashing uniforme independente equivalente a escolher uma funo hash uniformemente e aleatoriamente de uma famlia de todas as possveis $m^n$ funes hash, onde cada funo mapeia as $n$ chaves para os $m$ valores hash de maneira diferente.

11.3.3 Propriedades de hashing aleatrio

Seja $h$ uma funo escolhida aleatoriamente e independentemente de uma famlia de funes hash $H$. Dentre a vasta gama de propriedades na literatura, algumas interessantes so:

  • $H$ uniforme se para qualquer chave $k$ e slot $q$, a probabilidade de $h(k)=q$ for $1/m$.
  • $H$ universal se para quaiquer chaves distintas $k_1,k_2$, a probabilidade de $h(k_1)=h(k_2)$ for no mximo $1/m$.
  • $H$ $\epsilon$-universal se para quaiquer chaves distintas $k_1,k_2$, a probabilidade de $h(k_1)=h(k_2)$ for no mximo $\epsilon$. Portanto, uma famlia de funes universal tambm $1/m$-universal.

11.3.4 Projetando uma famlia universal de funes hash

Famlia universal de funes baseada em teoria dos nmeros

Comece escolhendo um nmero primo $p$ tal que qualquer chave $k$ esteja entre $0$ e $p-1$ inclusivos. Seja $Z_p=\{0,1,..., p-1\}$ e $Z^*_p=\{1,..., p-1\}$. Como o universo de chaves maior que o tamanho da tabela, ento $p\gt m$. O captulo 31 ensina como calcular resto da diviso de nmeros primos eficientemente.

Dados quaisquer $a\in Z^*_p$ e $b\in Z_p$, definimos a funo hash $h_{ab}$ como:

$$\begin{equation} h_{ab}(k)=((ak+b)\mod p) \mod m \end{equation}$$

Por exemplo, para $p=17$ e $m=6$, temos:

$h_{3,4}(8)=((3\cdot 8 + 4) \mod 17)\mod 6=(28\mod 17)\mod 6=11\mod 6 = 5$.

Dados $p,m$, a famlia de funes :

$$\begin{equation} H_{pm}=\{h_{ab}: a\in Z^*_p \text{ e } b\in Z_p\} \end{equation}$$

A vantagem desse mtodo que $m$ pode ser arbitrrio (no precisa ser primo). Como voc pode escolher entre $p-1$ valores de $a$ e $p$ valores de $b$, a famlia $H_{pm}$ contm $p(p-1)$ funes.

Hashing criptogrfico

Tambm podemos usar funes de hash criptogrfico. So projetadas para aplicaes com requisitos muito alm das necessidades de uma tabela hash. Mas por serem robustas e amplamente implementadas, podemos aplic-las aqui tambm.

Uma funo hash criptogrfica recebe como entrada um vetor de bytes arbitrrio e retorna uma sada de comprimento fixo. Por exemplo, a famosa funo SHA-256 produz uma sada de 256 bits (32 bytes) para qualquer entrada.

Inclusive podemos fazer proveito de alguns processadores que implementam instrues especficas para auxiliar no clculo de funes criptogrficas, como as instrues AES-NI que computam arredondamentos do algoritmo Advanced Encryption Standard (AES).

Dentre as maneiras de usar funes hash criptogrficas como funes hash simples, temos:

$h(k)=\text{SHA-256(k)} \mod m$.

Para definir uma famlia de funes hash, podemos adicionar uma string tempero (salt string) entrada antes de gerar seu hash:

$h_a(k)=\text{SHA-256(a k)} \mod m$,

onde $a k$ a concatenao de $a$ com $k$. A literatura sobre Message Authentication Codes (MACs) fornece abordagens adicionais.

Abordagens criptogrficas para o design de funes hash esto se tornando mais prticas medida que os computadores organizam suas memrias em hierarquias com capacidades e velocidades diferentes. A Seo 11.5 discute um design de funo hash baseado no mtodo de criptografia RC6.

Assim, suficiente mostrar que se a string $x$ pode ser derivada da string $y$ pela troca de um nico par de caracteres, ento $x$ e $y$ tm o mesmo hash.

    Vamos denotar o $i$-simo caracter de $x$ por $x_i$ e similarmente para $y$. A interpretao de $x$ no radix $2^p$ $\sum_{i=0}^{n-1}x_i2^{ip}$, e portanto, $h(x)=(\sum_{i=0}^{n-1}x_i2^{ip}) \mod (2^p-1)$.

    Suponha que $x$ e $y$ so strings idnticas de $n$ caracteres, exceto que os caracteres nas posies $a$ e $b$ so trocadas: $x_a=y_b$ e $y_a=x_b$. Sem perda de generalidade, seja $a\gt b$. Temos:

    $h(x)-h(y)=(\sum_{i=0}^{n-1}x_i2^{ip})\mod (2^p-1)-(\sum_{i=0}^{n-1}y_i2^{ip})\mod (2^p-1)$.

    Como $0\leq h(x),h(y) \lt 2^p-1$, temos que $-(2^p-1)\lt h(x)-h(y)\lt 2^p-1$. Se mostrarmos que $(h(x)-h(y))\mod(2^p-1)=0$, ento $h(x)=h(y)$.

    Como as somas nas funes hash so a mesma exceto para os ndices $a$ e $b$, temos:

    $(h(x)-h(y))\mod(2^p-1)$
    $=((x_a2^{ap}+x_b2^{bp})-(y_a2^{ap}+y_b2^{bp}))\mod(2^p-1)$
    $=((x_a2^{ap}+x_b2^{bp})-(x_b2^{ap}+x_a2^{bp}))\mod(2^p-1)$
    $=((x_a-x_b)(2^{ap}-2^{bp}))\mod(2^p-1)$
    $=((x_a-x_b)2^{bp}(2^{(a-b)p}-1))\mod(2^p-1)$

    Pela soma dos termos de uma PG, temos:

    $\sum_{i=0}^{a-b-1}2^{pi}=\dfrac{2^{(a-b)p}-1}{2^p-1}$,

    E multiplicando ambos os lados por $2^p-1$, obtemos:

    $2^{(a-b)p}-1=(\sum_{i=0}^{a-b-1}2^{pi})(2^p-1)$.

    Portanto,

    $(h(x)-h(y))\mod(2^p-1)$
    $=((x_a-x_b)2^{bp}(\sum_{i=0}^{a-b-1}2^{pi})(2^p-1))\mod(2^p-1)$
    $=0$,

    j que um dos fatores $2^p-1$. Assim, mostramos que $(h(x)-h(y))\mod(2^p-1)=0$, e portanto, $h(x)=h(y)$.

  5. Demonstre que qualquer famlia $$-universal $H$ de funes hash de um conjunto finito $U$ para um conjunto finito $Q$ possui $ \ge 1/|Q| 1/|U|$.

11.4 Endereamento aberto

No endereamento aberto, todos os elementos ficam na prpria tabela hash. Porm, a tabela hash pode ficar cheia. Seu fator de carga $\alpha$ nunca pode exceder 1.

O exerccio 11.2-4 do livro usa uma lista encadeada entre as prprias posies da tabela. Mas a vantagem do endereamento aberto que ele evita ponteiros por completo. A memria extra liberada por no armazenarmos ponteiros fornece tabela hash um nmero maior de posies para a mesma quantidade de memria, o que produz potencialmente menor nmero de colises e buscas mais rpida.

Para inserir com endereamento aberto, sondamos a tabela hash at encontrar uma posio vazia na qual inserir a chave. Para determinar quais sero as posies a sondar, adicionamos funo hash um segundo parmetro, o nmero da sondagem (a partir de 0).

hashInsert(T,k)
  i = 0
  while i < m
    j = h(k,i)
    if T[j] == NIL
      T[j] = k
      return j
    else i++
  error "hash table overflow"

Com endereamento aberto, exigimos que, para toda chave $k$, a sequncia de sondagem $\lang h(k,0), h(k,1), , h(k,m-1)\rang $ seja uma permutao de $\lang 0, 1, ..., m - 1\rang $, de modo que toda posio da tabela hash seja eventualmente considerada para uma nova chave, medida que a tabela preenchida.

O algoritmo $\text{hashSearch}$ sonda a mesma sequncia de posies que o algoritmo de insero para uma mesma chave $k$. Portanto, a busca pode terminar (sem sucesso) quando encontra uma posio vazia, j que $k$ teria sido inserido ali e no mais adiante em sua sequncia de sondagem. (Esse argumento supe que no h excluso de chaves na tabela hash.)

hashSearch(T,k)
    i = 0
    while i < m and T[j] ≠ NIL
        j = h(k,i)
        if T[j] == k
            return j
        i++
    return NIL

Quando exclumos uma chave da posio $i$, no podemos simplesmente marcar essa posio com $\text{NIL}$. Se fizssemos isso, poderamos no conseguir recuperar nenhuma chave $k$ que no foi inserida em $i$ por estar ocupada. Podemos resolver esse problema armazenando o valor especial $\text{DELETED}$ em vez de $\text{NIL}$. Ento, modificaramos o procedimento $\text{hashInsert}$ para tratar tal posio como se ela estivesse vazia, de modo a podermos inserir uma nova chave. No necessrio modificar $\text{hashSearch}$, j que ele passar por valores $\text{DELETED}$ enquanto estiver buscando. Entretanto, quando usamos o valor especial $\text{DELETED}$, os tempos de busca no dependem mais do fator de carga $\alpha$. Por essa razo, o encadeamento mais comumente selecionado como uma tcnica de resoluo de colises quando precisamos excluir chaves.

Num hashing uniforme, a sequncia de sondagem de cada chave tem igual probabilidade de ser qualquer uma das $m!$ permutaes de $\lang 0, 1, ..., m - 1\rang$. O hashing uniforme generaliza a noo de hashing uniforme independente definida anteriormente para uma funo hash que produz no apenas um nico nmero, mas toda uma sequncia de sondagem. Contudo, o verdadeiro hashing uniforme difcil de implementar e, na prtica, so usadas aproximaes adequadas (como o hashing duplo, definido a seguir).

Examinaremos trs tcnicas comumente utilizadas para calcular as sequncias de sondagem exigidas para o endereamento aberto: sondagem linear, sondagem quadrtica e hash duplo. O hash duplo tem o maior nmero de sequncias de sondagem e, como seria de esperar, parece dar os melhores resultados.

Sondagem linear

Dada uma funo hash comum $h': U \{0, 1, ..., m - 1\}$, qual nos referimos como uma funo hash auxiliar, o mtodo de sondagem linear usa a funo hash:

$h(k, i) = (h'(k) + i) \mod m$

para $i = 0, 1, ..., m - 1$.

A sondagem linear fcil de implementar, mas sofre de um problema conhecido como agrupamento primrio (primary clustering). Longas sequncias de posies ocupadas se acumulam, aumentando o tempo mdio de busca. Os clusters surgem porque uma posio vazia precedida por $i$ posies cheias a prxima a ser ocupada com probabilidade $(i + 1)/m$.

Outra desvantagem que a sonda inicial determina toda a sequncia apenas $m$ sequncias possveis.

Sondagem quadrtica

A sondagem quadrtica utiliza a funo:

$h(k, i) = (h'(k) + c_1i + c_2i^2) \mod m$,

onde $h'$ uma funo hash auxiliar, $c1$ e $c2$ so constantes positivas auxiliares e $i = 0, 1, ..., m - 1$.

Assim como a sondagem linear, a sequncia comea em $h'(k)$. Mas diferentemente da linear, a quadrtica pula pelas posies da tabela de acordo com a funo.

Esse mtodo funciona muito melhor que a sondagem linear mas, para fazer pleno uso da tabela hash, os valores de $c1$, $c2$ e $m$ so restritos.

Alm disso, se duas chaves tm a mesma posio inicial de sondagem, ento suas sequncias de sondagem so iguais, j que $h(k_1, 0) = h(k_2, 0)$ implica $h(k_1, i) = h(k_2, i)$. Essa propriedade conduz a uma forma mais branda de agrupamento, denominada agrupamento secundrio. Como na sondagem linear, a sondagem inicial determina a sequncia inteira; assim, apenas $m$ sequncias de sondagem distintas so utilizadas.

Hash duplo

O hash duplo oferece um dos melhores mtodos disponveis para endereamento aberto porque as permutaes produzidas tm muitas das caractersticas de permutaes escolhidas aleatoriamente. O hash duplo usa a funo:

$h(k, i) = (h_1(k) + ih_2(k))\mod m$,

onde $h_1$ e $h_2$ so funes hash auxiliares. Diferentemente do caso de sondagem linear ou quadrtica, aqui a sequncia de sondagem depende da chave $k$ de duas maneiras, j que tanto a posio inicial de sondagem quanto o deslocamento (offset) podem variar. No caso, $h_1$ define a sonda inicial e $h_2$ define as sondas seguintes. A Figura 11.5 d um exemplo de insero por hash duplo.

Na Figura 11.5, $h_1(k)=k\mod 13$ e $h_2(k)=1+(k\mod 11)$.

O valor $h_2(k)$ e o tamanho $m$ da tabela hash devem ser primos entre si para que a tabela hash inteira seja examinada (exerccio 11.4-4). Uma forma conveniente de assegurar essa condio que $m$ seja uma potncia de $2$ e projetar $h_2$ de modo que sempre retorne um nmero mpar (por que uma potncia de 2 e um mpar so primos entre si?). Outra maneira que $m$ seja primo e projetar $h_2$ de modo que sempre retorne um inteiro positivo menor que $m$. Por exemplo, poderamos escolher $m$ primo e fazer:

$h_1(k) = k\mod m$,
$h_2(k) = 1 + (k\mod m')$,

onde o valor de $m'$ escolhido ligeiramente menor que m (digamos, $m - 1$). Por exemplo, se $k = 123456$, $m = 701$ e $m' = 700$, temos $h_1(k) = 80$ e $h_2(k) = 257$; assim, primeiro sondamos a posio $80$ e depois examinamos cada $257$ posio (mdulo m) at encontrarmos a chave ou termos examinado todas as posies.

Quando $m$ primo ou uma potncia de $2$, o hash duplo melhor que a sondagem linear ou quadrtica no sentido de que so usadas $\Theta(m^2)$ sequncias de sondagem em vez de $\Theta(m)$, j que cada par $(h_1(k), h_2(k))$ possvel gera uma sequncia de sondagem distinta. O resultado que, para tais valores de $m$, o desempenho do hash duplo parece ficar bem prximo do esquema ideal do hashing uniforme.

Anlise do endereamento aberto

No endereamento aberto temos no mximo um elemento por posio e, portanto, $n \leq m$, o que implica $\alpha 1$.

Supomos que estamos usando hashing uniforme. Nesse esquema idealizado, a sequncia de sondagem $\lang h(k, 0), h(k, 1), ..., h(k, m - 1)\rang $ usada para inserir ou procurar cada chave $k$ tem igual probabilidade de ser qualquer permutao de $\lang 0, 1, ..., m - 1\rang$.

Teorema 11.6

Dada uma tabela hash com endereamento aberto e fator de carga $\alpha = n/m \lt 1$, o nmero esperado de sondagens em uma busca malsucedida no mximo $1/(1 - \alpha )$, supondo hashing uniforme.

Prova no livro.

Isso indica que se a tabela estiver metade cheia, o nmero mdio de sondagens numa busca mal-sucedida no mximo $1/(1-.5)=2$. Se estiver $90\%$ cheia, passa a ser $1/(1-.9)=10$.

Corolrio 11.7

Inserir um elemento em uma tabela hash de endereamento aberto com fator de carga $\alpha$ exige no mximo $1/(1 - \alpha )$ sondagens, em mdia, considerando hash uniforme.

Prova

Um elemento inserido somente se houver espao na tabela e, portanto, $\alpha < 1$. Inserir uma chave requer uma busca malsucedida seguida pela colocao da chave na primeira posio vazia encontrada. Assim, o nmero esperado de sondagens , no mximo, $1/(1 - \alpha )$.

Teorema 11.8

Dada uma tabela hash de endereamento aberto com fator de carga $\alpha < 1$, considerando hash uniforme e tambm que cada chave na tabela tem igual probabilidade de ser procurada, o nmero esperado de sondagens em uma busca bem sucedida , no mximo $(1/ \alpha ) \ln (1/(1- \alpha ))$

Se a tabela hash estiver metade cheia, o nmero esperado de sondagens em uma busca bem-sucedida ser menor que $1,387$. Se estiver $90\%$ cheia, passa a ser algo menor que $2,559$.
