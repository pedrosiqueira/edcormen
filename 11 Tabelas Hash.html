11 Tabelas hash

Muitas aplicações exigem um conjunto dinâmico que suporte somente as operações de dicionário INSERT, SEARCH e DELETE. Por exemplo, um compilador que traduz uma linguagem de programação mantém uma tabela de símbolos. O tempo médio para buscar um elemento em uma tabela hash, também chamada de tabela de espalhamento, ou tabela de dispersão, é $O(1)$.

11.1 Tabelas de endereçamento direto

O endereçamento direto é uma técnica simples que funciona bem quando o universo $U$ de chaves é razoavelmente pequeno. Suponha que uma aplicação necessite de um conjunto dinâmico no qual cada elemento tem uma chave extraída do universo $U = \{0, 1, ..., m - 1\}$, onde $m$ não é muito grande. Consideramos que não há dois elementos com a mesma chave.

Para representar o conjunto dinâmico, usamos um array, ou uma tabela de endereçamento direto, denotada por $T[0 .. m - 1]$, na qual cada posição (ou lacuna, slot) corresponde a uma chave no universo $U$. A Figura 11.1 ilustra a abordagem; a posição $k$ aponta para um elemento no conjunto com chave $k$. Se o conjunto não contém nenhum elemento com chave $k$, então $T[k] = NIL$.

A implementação das operações de dicionário é trivial:

directAddressSearch(T,k)
    return T[k]

directAddressInsert(T,x)
    T[x.key] = x

directAddressDelete(T,x)
    T[x.key] = NIL

Cada uma dessas operações custa somente $O(1)$.

Em vez de armazenar o objeto fora da tabela de endereçamento direto, podemos armazenar o objeto na própria tabela. Essa alternativa economizaria espaço? Usaríamos uma chave especial dentro do objeto para indicar uma posição vazia. Além disso, muitas vezes, é desnecessário armazenar a chave do objeto, já que, se temos o índice de um objeto na tabela, temos sua chave.

11.2 Tabelas hash

Se o universo $U$ é grande, armazenar uma tabela $T$ de tamanho $|U|$ pode ser impraticável ou mesmo impossível. Além disso, o conjunto $K$ de chaves realmente armazenadas pode ser tão pequeno em relação a $U$ que grande parte do espaço alocado para $T$ seria desperdiçada.

Com tabela de endereçamento direto, um elemento com a chave $k$ é armazenado na posição $k$. Com tabela hash (tabela de espalhamento, ou tabela de dispersão), esse elemento é armazenado na posição $h(k)$; isto é, usamos uma função hash $h$ para calcular a posição da chave $k$.

A Figura 11.2 ilustra a ideia básica:

Após o hash, duas chaves podem ser mapeadas para a mesma posição. Chamamos essa situação de colisão. Podemos evitar colisão escolhendo uma função hash adequada. Uma ideia é fazer $h$ parecer “aleatória”.

Seja $m$ o tamanho da tabela hash. Como $|U|>m$, é impossível evitar totalmente colisões.

Hashing uniform independente

Para cada elemento $k$ do domínio $U$, uma função hash $h$ ideal geraria o mesmo valor $h(k)$ independentemente e uniformemente aleatório do intervalo $\{0, 1, ..., m - 1\}$. Chamamos tal função de função hash uniforme independente. Na prática, temos aproximações desse ideal.

Resolução de colisões por encadeamento

No encadeamento, todos os elementos resultantes do hash vão para a mesma posição em uma lista encadeada, como mostra a Figura 11.3.

A seguir, a implementação das operações:

chainedHashInsert(T,x)
    insert x at the head of list T[h(x.key)]

chainedHashSearch(T,k)
    search for an element with key k in list T[h(k)]

chainedHashDelete(T,x)
    delete x from the list T[h(x.key)]

Se a lista for duplamente encadeada, a operação de exclusão pode ser feita em tempo $O(1)$, já que o argumento da função não é a chave do elemento a ser excluído, e sim o ponteiro para o elemento. A inserção é $O(1)$.

Análise do hashing com encadeamento

Dada uma tabela hash $T$ com $m$ posições que armazena $n$ elementos, definimos o fator de carga $\alpha$ de $T$ como $n/m$, isto é, o número médio de elementos armazenados em uma lista encadeada.

O pior caso do hashing com encadeamento é terrível: após o hashing, todas as $n$ chaves vão para a mesma posição, criando uma lista de comprimento $n$.

O desempenho do hashing para o caso médio depende de como a função hash $h$ distribui as chaves entre as $m$ posições, em média. Por enquanto, vamos considerar hashing uniform independente, isto é, qualquer elemento tem igual probabilidade de cair em qualquer das $m$ posições, independente dos elementos já na tabela.

Para $j = 0, 1, ..., m - 1$, vamos denotar o comprimento da lista $T[j]$ por $n_j$, de modo que

$$\begin{equation} n=n_0+n_1+...+n_{m-1} \end{equation}$$

e o valor esperado de $n_j$ é $E[n_j] = \alpha = n/m$.

Assumimos que o custo de calcular o valor hash $h(k)$ é $O(1)$. Assim, o tempo para procurar por um elemento com chave $k$ depende do comprimento $n_{h(k)}$ da lista $T[h(k)]$.

Teorema 11.1

Em uma tabela hash com colisões resolvidas com encadeamento, uma busca malsucedida demora tempo médio $\Theta(1+\alpha)$ com hashing uniforme independente.

Prova

Qualquer chave $k$ ainda não armazenada na tabela tem igual probabilidade de ocupar qualquer das $m$ posições após o hash. O tempo esperado para procurar sem sucesso uma chave $k$ é o tempo esperado para buscar até o fim da lista $T[h(k)]$, que tem o comprimento esperado $E[n_{h(k)}] = \alpha$. Assim, o número esperado de elementos examinados em uma busca malsucedida é $\alpha$, e o tempo total exigido (incluindo o tempo para se calcular $h(k)$) é $\Theta(1 + \alpha)$.

Teorema 11.2

Em uma tabela hash com colisões resolvidas com encadeamento, uma busca bem sucedida demora tempo médio $\Theta(1+\alpha)$ com hashing uniforme independente.

Prova

A situação para uma busca bem-sucedida é ligeiramente diferente, pois o elemento desejado pode ser qualquer um dos elementos presentes na tabela, de modo que quanto maior uma lista, maior a probabilidade da busca cair nela. Todavia, o tempo de busca esperado ainda é $\Theta(1 + \alpha)$. Mais detalhes sobre a prova encontra-se no livro.

O que estas análises significam? Se o número de posições da tabela hash é no mínimo proporcional ao número de elementos na tabela, temos $n = O(m)$ e, consequentemente, $\alpha = n/m = O(m)/m = O(1)$. Assim, a busca demora tempo constante na média.

11.3 Funções hash

Uma boa função hash satisfaz (aproximadamente) a premissa do hashing uniforme independente: cada chave tem igual probabilidade de cair em qualquer das $m$ posições, independentemente da posição que qualquer outra chave ocupa. Infelizmente, normalmente não temos nenhum meio de verificar essa condição, já que raramente conhecemos a distribuição de probabilidade da qual as chaves são extraídas. Além disso, as chaves poderiam não ser extraídas independentemente. Ocasionalmente conhecemos a distribuição. Por exemplo, se soubermos que as chaves são números reais aleatórios $k$, independente e uniformemente distribuídos na faixa $0 \leq k \lt 1$, então a função hash $h(k)=\lfloor km \rfloor$ satisfaz a condição de hashing uniforme independente.

A maior parte das funções hash considera como universo de chaves o conjunto $N = \{0, 1, 2, ...\}$ de números naturais. Assim, se as chaves não forem números naturais, temos de encontrar um modo de interpretá-las como números naturais. Por exemplo, o conjunto de caracteres ASCII possui 128 possíveis caracteres. Para converter uma string ASCII (dígitos na base 128) em um número natural (dígitos na base 10), usamos a fórmula de conversão de radix. Assim, a string “pt” (caracteres 112 e 116) na notação decimal seria 112*128+126=14452. Como seria teu nome na notação decimal? Portanto, consideramos a partir daqui que as chaves são números naturais.

11.3.1 Hashing estático

O método da divisão

Função hash: $h(k) = k \mod m$.

Por exemplo, para tabela hash com $m = 12$ e chave $k = 100$, então $h(k) = 4$.

Vantagem: rapidez, pois o cálculo envolve apenas uma operação de divisão.

Alguns valores para $m$ são ruins, como uma potência de $2$, já que, se $m = 2^p$, então $h(k)$ usará apenas os $p$ bits menos significativos de $k$ para o cálculo. É melhor garantir que a função hash dependa de todos os bits da chave.

Um primo não muito próximo de uma potência de $2$ é uma boa escolha para $m$. Por exemplo, considere uma tabela hash com colisões resolvidas por encadeamento que armazena $n=2000$ strings. Um bom tamanho para essa tabela é $701$, pois é um primo não próximo de uma potência de $2$ e permite uma média de três elementos colididos ($2000/3 ≈ 701$). A função hash seria $h(k)=k \mod 701$.

O método da multiplicação

Função hash:

$$h(k) = \lfloor m (k A \mod 1)\rfloor$$

que pode ser lida da seguinte forma:

  1. Escolha uma constante $A$ no intervalo $0 \lt A \lt 1$
  2. Multiplique a chave $k$ pela constante $A$
  3. Extraia a parte fracionária de $k A$, isto é, $(k A \mod 1)$ é o mesmo que $(k A - ⌊k A⌋)$
  4. Multiplique a parte fracionária por $m$
  5. Pegue o piso do resultado

Vantagem: o valor de $m$ não é crítico. Entretanto, este método não é tão rápido quanto o da divisão. Em geral, $m = 2^p$, para algum inteiro $p$, já que podemos implementar facilmente a função na maioria dos computadores (mais informações no livro).

Alguns valores para $A$ funcionam melhor que outros. A escolha ótima depende das características dos dados aos quais o hash está sendo aplicado. Knuth sugere que $A \approx ( \sqrt{5} - 1 )/2=0.6180339887...$. funciona bem.

Como exemplo, suponha $k=123456$, $p=14$, $m=2^p=16384$. Assim, $kA = 76300.004115107$. Então, $h(k) = ⌊m * 0.004115107⌋ = 67$.

O método da multiplicação com deslocamento

Se as chaves couberem em uma palavra de máquina de $w$ bits, e $m=2^l\le w$, podemos usar o método da multiplicação com deslocamento (multiply-shift).

Seja a constante $a=A2^w$, onde $0\le A \le 1$, de modo que $0\le a \le 2^w$, a função hash é:

$$\begin{equation}h_a(k) = (ka \mod 2^w) \gg (w-l)\end{equation}$$

que pode ser lida assim:

  1. Multiplique $k$ pelo inteiro $a$ de $w$ bits. O resultado é um valor com $2w$ bits.
  2. Pegue o resto da divisão por $2^w$, eliminando os $w$ bits mais significativos ($r_1$), deixando apenas o $w$ bits menos significativos ($r_0$).
  3. Desloque (shift) à direita $w-l$ bits, deixando apenas os $l$ bits mais significativos de $r_0$, que é seu valor hash.

Esse método pode ser implementado com três instruções de máquina simples: multiplicação, subtração e deslocamento à direita.

Como exemplo, suponha $k=123456$, $l=14$, $m=2^l=16384$, $w=32$ e $a=2654435769$, conforme sugestão de Knuth. Assim, $ka = 76300\cdot 2^{32} + 17612864$, sendo $r_1=76300$ e $r_0=17612864$. Os 14 bits mais significativos de $r_0$ são $h_a(k)=67$.

11.3.2 Hashing aleatório

Suponha que um cracker escolha chaves sabendo a função hash estática implementada. Ele pode escolher $n$ chaves que caiam no mesmo slot, resultando em um tempo médio de busca de $\Theta(n)$. Qualquer função hash estática é vulnerável a esse ataque. A única maneira eficaz de evitar essa situação é escolher a função hash aleatoriamente e independentemente das chaves. Essa abordagem é chamada de hashing aleatório. Um caso especial dessa abordagem, chamado de hashing universal, pode proporcionar um desempenho médio comprovadamente bom, independentemente das chaves escolhidas pelo cracker.

Para usar hashing aleatório, no início da execução do programa, escolha aleatoriamente uma função hash de uma família adequada de funções. Assim como no quicksort, aleatoriedade garante que uma sequência de entrada não invoque apenas o pior caso a cada execução.

Por exemplo, uma variante aleatória simples do método de multiplicação com deslocamento que funciona bem na média é, no início do programa, escolher aleatoriamente um número ímpar para o valor de $a$.

Uma família $H$ de funções hash é universal se, ao escolher aleatoriamente uma função hash $h$, a chance de colisão entre duas chaves arbitrárias distintas $k_1,k_2$ for no máximo a chance $1/m$ da colisão se $h(k_1)$ e $h(k_2)$ fossem valores aleatoriamente e independentemente escolhidos entre $\{0,1,..., m-1\}$.

Hashing uniforme independente é equivalente a escolher uma função hash uniformemente e aleatoriamente de uma família de todas as possíveis $m^n$ funções hash, onde cada função mapeia as $n$ chaves para os $m$ valores hash de maneira diferente.

Corolário 11.3

Usando hashing universal e resolução de colisão por encadeamento em uma tabela inicialmente vazia de tamanho $m$, leva tempo $\Theta(s)$ para lidar com $s$ operações de inserção, exclusão e busca, contendo $n=O(m)$ operações de inserção.

Prova: No livro.

11.3.3 Propriedades de hashing aleatório

Seja $h$ uma função escolhida aleatoriamente e independentemente de uma família de funções hash $H$. Dentre a vasta gama de propriedades na literatura, algumas interessantes são:

  • $H$ é uniforme se para qualquer chave $k$ e slot $q$, a probabilidade de $h(k)=q$ for $1/m$.
  • $H$ é universal se para quaiquer chaves distintas $k_1,k_2$, a probabilidade de $h(k_1)=h(k_2)$ for no máximo $1/m$.
  • $H$ é $\epsilon$-universal se para quaiquer chaves distintas $k_1,k_2$, a probabilidade de $h(k_1)=h(k_2)$ for no máximo $\epsilon$. Portanto, uma família de funções universal também é $1/m$-universal.

11.3.4 Projetando uma família universal de funções hash

Família universal de funções baseada em teoria dos números

Comece escolhendo um número primo $p$ tal que qualquer chave $k$ esteja entre $0$ e $p-1$ inclusivos. Seja $Z_p=\{0,1,..., p-1\}$ e $Z^*_p=\{1,..., p-1\}$. Como o universo de chaves é maior que o tamanho da tabela, então $p\gt m$. O capítulo 31 ensina como calcular resto da divisão de números primos eficientemente.

Dados quaisquer $a\in Z^*_p$ e $b\in Z_p$, definimos a função hash $h_{ab}$ como:

$$\begin{equation} h_{ab}(k)=((ak+b)\mod p) \mod m \end{equation}$$

Por exemplo, para $p=17$ e $m=6$, temos:

$h_{3,4}(8)=((3\cdot 8 + 4) \mod 17)\mod 6=(28\mod 17)\mod 6=11\mod 6 = 5$.

Dados $p,m$, a família de funções é:

$$\begin{equation} H_{pm}=\{h_{ab}: a\in Z^*_p \text{ e } b\in Z_p\} \end{equation}$$

A vantagem desse método é que $m$ pode ser arbitrário (não precisa ser primo). Como você pode escolher entre $p-1$ valores de $a$ e $p$ valores de $b$, a família $H_{pm}$ contém $p(p-1)$ funções.

Teorema 11.4

A família $H_{pm}$ é universal.

Prova: No livro.

Família $2/m$-universal de funções baseada no método de multiplicação com deslocamento

Essa é a família recomendada pelo autor, sendo excepcionalmente eficiente e $2/m$-universal. É definida como:

$$\begin{equation} H=\{h_a:a\text{ é ímpar, }1\le a \lt m \text{ e } h_a \text{ é definida pela equação (11.2)}\} \end{equation}$$

Teorema 11.5

A família de funções hash $H$ da equação (11.5) é $2/m$-universal.

Isto é, a probabilidade de duas chaves distintas colidirem é no máximo $2/m$. Em muitas situações práticas, comparada à função hash universal, a velocidade de computação de valores hash mais do que compensa pelo limite superior maior de probabilidade de colisão.

11.3.5 Hashing de longas entradas, como vetores ou strings

Às vezes, a entrada de uma função hash pode ser tão longa que não pode ser facilmente codificada com um número primo $p$ de tamanho razoável ou codificada em uma única palavra, por exemplo, de 64 bits. Como exemplo, considere vetores de bytes (que é como strings em muitas linguagens de programação são armazenadas). Um vetor pode ter um comprimento não negativo arbitrário, de modo que a função hash pode receber entradas de comprimentos variados.

Abordagens número-teóricas

Hashing criptográfico

Também podemos usar funções de hash criptográfico. São projetadas para aplicações com requisitos muito além das necessidades de uma tabela hash. Mas por serem robustas e amplamente implementadas, podemos aplicá-las aqui também.

Uma função hash criptográfica recebe como entrada um vetor de bytes arbitrário e retorna uma saída de comprimento fixo. Por exemplo, a famosa função SHA-256 produz uma saída de 256 bits (32 bytes) para qualquer entrada.

Inclusive podemos fazer proveito de alguns processadores que implementam instruções específicas para auxiliar no cálculo de funções criptográficas, como as instruções “AES-NI” que computam arredondamentos do algoritmo Advanced Encryption Standard (AES).

Dentre as maneiras de usar funções hash criptográficas como funções hash simples, temos:

$h(k)=\text{SHA-256(k)} \mod m$.

Para definir uma família de funções hash, podemos adicionar uma string “tempero” (salt string) à entrada antes de gerar seu hash:

$h_a(k)=\text{SHA-256(a ‖ k)} \mod m$,

onde $a ‖ k$ é a concatenação de $a$ com $k$. A literatura sobre Message Authentication Codes (MACs) fornece abordagens adicionais.

Abordagens criptográficas para o design de funções hash estão se tornando mais práticas à medida que os computadores organizam suas memórias em hierarquias com capacidades e velocidades diferentes. A Seção 11.5 discute um design de função hash baseado no método de criptografia RC6.

  2. Suponha que geremos hash de uma string de $r$ caracteres em m slots, tratando-a como um número de radix (base) $128$ e usando o método da divisão. Podemos facilmente representar o número $m$ como uma palavra de computador de $32$ bits. Porém, a string $r$, tratada como um número de radix 128, requer muitas palavras de computador. Como podemos aplicar o método da divisão para calcular o hash da string usando um número constante de palavras além da própria string?

    3. Cada caractere ocupa 7 bits (128 possibilidades). Logo, uma string com 5 caracteres já extrapola o tamanho de uma palavra de computador.

    Uma abordagem para resolver esse problema é usar um hash contínuo (rolling hash). A ideia é calcular o valor do hash de forma incremental, atualizando-o à medida que cada caractere é adicionado, em vez de processar a string inteira de uma vez. Isso evita a necessidade de armazenar a string inteira usando várias palavras de computador.

    Veja como funciona (em python):

    def rollingHash(s,m):
    sum = 0
    for c in s:
        sum = (sum * 128 + ord(c)) % m
    return sum

    A cada etapa, $sum$ será o hash do prefixo da string processada até o momento e sempre caberá em uma única palavra de 32 bits.

    Este algoritmo custa um tempo $O(r)$, onde $r$ é o comprimento da string, e custa um espaço $O(1)$, pois requer apenas uma quantidade constante de armazenamento além da própria string.

    Observe que o hash gerado por essa técnica possui o mesmo valor do hash da técnica padrão:

    def hash(s,m):
    r = len(s)
    sum = 0
    for i in range(0,r):
        sum+=ord(s[i])*128**(r-1-i)
    return sum % m

Mostre que se pudermos derivar a string $x$ da string $y$ permutando seus caracteres, então $x$ e $y$ possuem o mesmo hash. Dê um exemplo de aplicação no qual essa propriedade seria indesejável em uma função hash.

    4. Primeiro, observamos que podemos gerar qualquer permutação por uma sequência de trocas de pares de caracteres. Pode-se provar essa propriedade formalmente, mas informalmente, considere que o quicksort funciona permutando pares de elementos e que é capaz de produzir qualquer permutação de seu array de entrada. Assim, é suficiente mostrar que se a string $x$ pode ser derivada da string $y$ pela troca de um único par de caracteres, então $x$ e $y$ têm o mesmo hash.

    Vamos denotar o $i$-ésimo caracter de $x$ por $x_i$ e similarmente para $y$. A interpretação de $x$ no radix $2^p$ é $\sum_{i=0}^{n-1}x_i2^{ip}$, e portanto, $h(x)=(\sum_{i=0}^{n-1}x_i2^{ip}) \mod (2^p-1)$.

    Suponha que $x$ e $y$ são strings idênticas de $n$ caracteres, exceto que os caracteres nas posições $a$ e $b$ são trocadas: $x_a=y_b$ e $y_a=x_b$. Sem perda de generalidade, seja $a\gt b$. Temos:

    $h(x)-h(y)=(\sum_{i=0}^{n-1}x_i2^{ip})\mod (2^p-1)-(\sum_{i=0}^{n-1}y_i2^{ip})\mod (2^p-1)$.

    Como $0\leq h(x),h(y) \lt 2^p-1$, temos que $-(2^p-1)\lt h(x)-h(y)\lt 2^p-1$. Se mostrarmos que $(h(x)-h(y))\mod(2^p-1)=0$, então $h(x)=h(y)$.

    Como as somas nas funções hash são a mesma exceto para os índices $a$ e $b$, temos:

    $(h(x)-h(y))\mod(2^p-1)$
    $=((x_a2^{ap}+x_b2^{bp})-(y_a2^{ap}+y_b2^{bp}))\mod(2^p-1)$
    $=((x_a2^{ap}+x_b2^{bp})-(x_b2^{ap}+x_a2^{bp}))\mod(2^p-1)$
    $=((x_a-x_b)(2^{ap}-2^{bp}))\mod(2^p-1)$
    $=((x_a-x_b)2^{bp}(2^{(a-b)p}-1))\mod(2^p-1)$

    Pela soma dos termos de uma PG, temos:

    $\sum_{i=0}^{a-b-1}2^{pi}=\dfrac{2^{(a-b)p}-1}{2^p-1}$,

    E multiplicando ambos os lados por $2^p-1$, obtemos:

    $2^{(a-b)p}-1=(\sum_{i=0}^{a-b-1}2^{pi})(2^p-1)$.

    Portanto,

    $(h(x)-h(y))\mod(2^p-1)$
    $=((x_a-x_b)2^{bp}(\sum_{i=0}^{a-b-1}2^{pi})(2^p-1))\mod(2^p-1)$
    $=0$,

    já que um dos fatores é $2^p-1$. Assim, mostramos que $(h(x)-h(y))\mod(2^p-1)=0$, e portanto, $h(x)=h(y)$.

  4. Considere uma tabela hash de tamanho $m = 1000$ e uma função hash $h(k)=⌊m (k A \mod 1)⌋$ para $A = (\sqrt{5}-1)/ 2$. Calcule as posições para as quais são mapeadas as chaves $61, 62, 63, 64, 65$.
  5. Demonstre que qualquer família $ϵ$-universal $H$ de funções hash de um conjunto finito $U$ para um conjunto finito $Q$ possui $ϵ \ge 1/|Q| − 1/|U|$.
  6. Seja $U$ o conjunto de $d$-tuplas de valores retirados de $\mathbb Z_p$, e seja $Q = \mathbb Z_p$, onde $p$ é primo. Definimos a função hash $h_b: U \rightarrow Q$ para $b \in \mathbb Z_p$ em uma entrada $d$-tupla $\langle a_0, a_1, \ldots, a_{d - 1} \rangle$ de $U$ como

    $$h_b(\langle a_0, a_1, \ldots, a_{d - 1} \rangle) = \Bigg(\sum_{j = 0}^{d - 1} a_jb^j \Bigg) \mod p,$$

    e seja $\mathcal{H} = \{h_b : b \in \mathbb Z_p\}$. Argumente que $\mathcal H$ é $\epsilon$-universal para $\epsilon=((d - 1) / p)$. ($\textit{Dica:}$ Veja o Exercício 31.4-4.)

11.4 Endereçamento aberto

No endereçamento aberto, todos os elementos ficam na própria tabela hash. Porém, a tabela hash pode “ficar cheia”. Seu fator de carga $\alpha$ nunca pode exceder 1.

O exercício 11.2-4 do livro usa uma lista encadeada entre as próprias posições da tabela. Mas a vantagem do endereçamento aberto é que ele evita ponteiros por completo. A memória extra liberada por não armazenarmos ponteiros fornece à tabela hash um número maior de posições para a mesma quantidade de memória, o que produz potencialmente menor número de colisões e buscas mais rápida.

Para inserir com endereçamento aberto, sondamos a tabela hash até encontrar uma posição vazia na qual inserir a chave. Para determinar quais serão as posições a sondar, adicionamos à função hash um segundo parâmetro, o número da sondagem (a partir de 0).

hashInsert(T,k)
  i = 0
  while i < m
    j = h(k,i)
    if T[j] == NIL
      T[j] = k
      return j
    else i++
  error "hash table overflow"

Com endereçamento aberto, exigimos que, para toda chave $k$, a sequência de sondagem $\lang h(k,0), h(k,1), …, h(k,m-1)\rang $ seja uma permutação de $\lang 0, 1, ..., m - 1\rang $, de modo que toda posição da tabela hash seja eventualmente considerada para uma nova chave, à medida que a tabela é preenchida.

O algoritmo $\text{hashSearch}$ sonda a mesma sequência de posições que o algoritmo de inserção para uma mesma chave $k$. Portanto, a busca pode terminar (sem sucesso) quando encontra uma posição vazia, já que $k$ teria sido inserido ali e não mais adiante em sua sequência de sondagem. (Esse argumento supõe que não há exclusão de chaves na tabela hash.)

hashSearch(T,k)
    i = 0
    while i < m and T[j] ≠ NIL
        j = h(k,i)
        if T[j] == k
            return j
        i++
    return NIL

Quando excluímos uma chave da posição $i$, não podemos simplesmente marcar essa posição com $\text{NIL}$. Se fizéssemos isso, poderíamos não conseguir recuperar nenhuma chave $k$ que não foi inserida em $i$ por estar ocupada. Podemos resolver esse problema armazenando o valor especial $\text{DELETED}$ em vez de $\text{NIL}$. Então, modificaríamos o procedimento $\text{hashInsert}$ para tratar tal posição como se ela estivesse vazia, de modo a podermos inserir uma nova chave. Não é necessário modificar $\text{hashSearch}$, já que ele passará por valores $\text{DELETED}$ enquanto estiver buscando. Entretanto, quando usamos o valor especial $\text{DELETED}$, os tempos de busca não dependem mais do fator de carga $\alpha$. Por essa razão, o encadeamento é mais comumente selecionado como uma técnica de resolução de colisões quando precisamos excluir chaves.

Num hashing uniforme, a sequência de sondagem de cada chave tem igual probabilidade de ser qualquer uma das $m!$ permutações de $\lang 0, 1, ..., m - 1\rang$. O hashing uniforme generaliza a noção de hashing uniforme independente definida anteriormente para uma função hash que produz não apenas um único número, mas toda uma sequência de sondagem. Contudo, o verdadeiro hashing uniforme é difícil de implementar e, na prática, são usadas aproximações adequadas (como o hashing duplo, definido a seguir).

Examinaremos três técnicas comumente utilizadas para calcular as sequências de sondagem exigidas para o endereçamento aberto: sondagem linear, sondagem quadrática e hash duplo. O hash duplo tem o maior número de sequências de sondagem e, como seria de esperar, parece dar os melhores resultados.

Sondagem linear

Dada uma função hash comum $h': U → \{0, 1, ..., m - 1\}$, à qual nos referimos como uma função hash auxiliar, o método de sondagem linear usa a função hash:

$h(k, i) = (h'(k) + i) \mod m$

para $i = 0, 1, ..., m - 1$.

A sondagem linear é fácil de implementar, mas sofre de um problema conhecido como agrupamento primário (primary clustering). Longas sequências de posições ocupadas se acumulam, aumentando o tempo médio de busca. Os clusters surgem porque uma posição vazia precedida por $i$ posições cheias é a próxima a ser ocupada com probabilidade $(i + 1)/m$.

Outra desvantagem é que a sonda inicial determina toda a sequência ⇒ apenas $m$ sequências possíveis.

Sondagem quadrática

A sondagem quadrática utiliza a função:

$h(k, i) = (h'(k) + c_1i + c_2i^2) \mod m$,

onde $h'$ é uma função hash auxiliar, $c1$ e $c2$ são constantes positivas auxiliares e $i = 0, 1, ..., m - 1$.

Assim como a sondagem linear, a sequência começa em $h'(k)$. Mas diferentemente da linear, a quadrática pula pelas posições da tabela de acordo com a função.

Esse método funciona muito melhor que a sondagem linear mas, para fazer pleno uso da tabela hash, os valores de $c1$, $c2$ e $m$ são restritos.

Além disso, se duas chaves têm a mesma posição inicial de sondagem, então suas sequências de sondagem são iguais, já que $h(k_1, 0) = h(k_2, 0)$ implica $h(k_1, i) = h(k_2, i)$. Essa propriedade conduz a uma forma mais branda de agrupamento, denominada agrupamento secundário. Como na sondagem linear, a sondagem inicial determina a sequência inteira; assim, apenas $m$ sequências de sondagem distintas são utilizadas.

Hash duplo

O hash duplo oferece um dos melhores métodos disponíveis para endereçamento aberto porque as permutações produzidas têm muitas das características de permutações escolhidas aleatoriamente. O hash duplo usa a função:

$h(k, i) = (h_1(k) + ih_2(k))\mod m$,

onde $h_1$ e $h_2$ são funções hash auxiliares. Diferentemente do caso de sondagem linear ou quadrática, aqui a sequência de sondagem depende da chave $k$ de duas maneiras, já que tanto a posição inicial de sondagem quanto o deslocamento (offset) podem variar. No caso, $h_1$ define a sonda inicial e $h_2$ define as sondas seguintes. A Figura 11.5 dá um exemplo de inserção por hash duplo.

Na Figura 11.5, $h_1(k)=k\mod 13$ e $h_2(k)=1+(k\mod 11)$.

O valor $h_2(k)$ e o tamanho $m$ da tabela hash devem ser primos entre si para que a tabela hash inteira seja examinada (exercício 11.4-4). Uma forma conveniente de assegurar essa condição é que $m$ seja uma potência de $2$ e projetar $h_2$ de modo que sempre retorne um número ímpar (por que uma potência de 2 e um ímpar são primos entre si?). Outra maneira é que $m$ seja primo e projetar $h_2$ de modo que sempre retorne um inteiro positivo menor que $m$. Por exemplo, poderíamos escolher $m$ primo e fazer:

$h_1(k) = k\mod m$,
$h_2(k) = 1 + (k\mod m')$,

onde o valor de $m'$ escolhido é ligeiramente menor que m (digamos, $m - 1$). Por exemplo, se $k = 123456$, $m = 701$ e $m' = 700$, temos $h_1(k) = 80$ e $h_2(k) = 257$; assim, primeiro sondamos a posição $80$ e depois examinamos cada $257$ª posição (módulo m) até encontrarmos a chave ou termos examinado todas as posições.

Quando $m$ é primo ou uma potência de $2$, o hash duplo é melhor que a sondagem linear ou quadrática no sentido de que são usadas $\Theta(m^2)$ sequências de sondagem em vez de $\Theta(m)$, já que cada par $(h_1(k), h_2(k))$ possível gera uma sequência de sondagem distinta. O resultado é que, para tais valores de $m$, o desempenho do hash duplo parece ficar bem próximo do esquema “ideal” do hashing uniforme.

Análise do endereçamento aberto

No endereçamento aberto temos no máximo um elemento por posição e, portanto, $n \leq m$, o que implica $\alpha ≤ 1$.

Supomos que estamos usando hashing uniforme. Nesse esquema idealizado, a sequência de sondagem $\lang h(k, 0), h(k, 1), ..., h(k, m - 1)\rang $ usada para inserir ou procurar cada chave $k$ tem igual probabilidade de ser qualquer permutação de $\lang 0, 1, ..., m - 1\rang$.

Teorema 11.6

Dada uma tabela hash com endereçamento aberto e fator de carga $\alpha = n/m \lt 1$, o número esperado de sondagens em uma busca malsucedida é no máximo $1/(1 - \alpha )$, supondo hashing uniforme.

Prova no livro.

Isso indica que se a tabela estiver metade cheia, o número médio de sondagens numa busca mal-sucedida é no máximo $1/(1-.5)=2$. Se estiver $90\%$ cheia, passa a ser $1/(1-.9)=10$.

Corolário 11.7

Inserir um elemento em uma tabela hash de endereçamento aberto com fator de carga $\alpha$ exige no máximo $1/(1 - \alpha )$ sondagens, em média, considerando hash uniforme.

Prova

Um elemento é inserido somente se houver espaço na tabela e, portanto, $\alpha < 1$. Inserir uma chave requer uma busca malsucedida seguida pela colocação da chave na primeira posição vazia encontrada. Assim, o número esperado de sondagens é, no máximo, $1/(1 - \alpha )$.

Teorema 11.8

Dada uma tabela hash de endereçamento aberto com fator de carga $\alpha < 1$, considerando hash uniforme e também que cada chave na tabela tem igual probabilidade de ser procurada, o número esperado de sondagens em uma busca bem sucedida é, no máximo $(1/ \alpha ) \ln (1/(1- \alpha ))$

Prova no livro.

Se a tabela hash estiver metade cheia, o número esperado de sondagens em uma busca bem-sucedida será menor que $1,387$. Se estiver $90\%$ cheia, passa a ser algo menor que $2,559$.
